# 数据库设计：去掉学校管理（默认武汉大学）

**需求来源：** `docs/remove_school_manage/PRD分析.md`、`技术方案.md`、`API设计.md`  
**功能短名：** `remove_school_manage`  
**目标库：** MySQL 8（与本仓库 `application.yml` / JPA `ddl-auto=update` 对齐）  
**设计原则：** **不删表、不删 `school_id` 列**；学校降为内部默认归属键；用轻量约束加固「一名一校 / 一模版一份报告」；历史多校数据**不强制迁移**（MVP 应用层过滤即可）。

|---|------|------|----------|

| Q6 | 报告唯一性 | 同模版仅一份未删 | `managed_report` 唯一约束从「校+模版」改为「模版」维度 |

## 1. 范围与实体

| 实体 | 表名 | 本期动作 |
|------|------|----------|
| 学校 | `school` | **改**：`name` 加唯一；语义变为「内部默认校容器」（产品不再 CRUD） |

| 报告实例 | `managed_report` | **改唯一约束**：未删时按 `template_id` 唯一；**保留** `school_id` 列 |
| 模版等 | `report_template*` | **不改** |

**本期新建表：** 无。

**本期不建 / 不做：**

| 项 | 原因 |
|----|------|
| 删除 `school` / 去掉各表 `school_id` | 仍作归属与聚合过滤键；技术方案明确后置 |

| 默认校配置表 / 系统参数表 | 校名写死，无需配置表 |
| 物理外键 CASCADE | 与仓库惯例一致：无 DB FK |

## 2. ER 关系（变更后语义）

```text
school（通常仅 1 条「武汉大学」，允许多余历史行残留）
  │
  ├── 1 ─── * grad_employment_record   （新数据 school_id = 默认校）
  └── 1 ─── * managed_report           （新/校正后 school_id = 默认校）

report_template 1 ─── * managed_report
  └── 未删报告：同一 template_id 至多 1 条（DB + 应用）
```

逻辑关系（**无物理外键**）：

- `grad_employment_record.school_id` → `school.id`
- `managed_report.school_id` → `school.id`
- `managed_report.template_id` → `report_template.id`

## 3. 表结构变更明细

### 3.1 `school` — 增量

**现有列（不变）：**

| 列名 | 类型 | 空 | 说明 |
|------|------|----|------|
| `id` | `BIGINT` | N | AI 主键；**不**承诺固定为某值 |
| `name` | `VARCHAR(128)` | N | 学校名称；默认校必须为精确字符串 `武汉大学` |
| `create_time` | `DATETIME` | N | |
| `update_time` | `DATETIME` | N | |

**新增约束：**

| 名 | 类型 | 说明 |
|----|------|------|
| `uk_school_name` | `UNIQUE (name)` | 防止 `getOrCreate` 并发插入同名多行 |

**加唯一前数据检查：** 若已存在重复 `name`，须先合并/改名再执行 DDL（见 §7.1）。

**种子数据（推荐，非强制）：** 若不存在名为「武汉大学」的行，插入一条。应用层 `DefaultSchoolService.getOrCreate()` 仍是主路径；种子仅降低冷启动空窗。

```sql
INSERT INTO school (name, create_time, update_time)
SELECT '武汉大学', NOW(), NOW()
FROM DUAL
WHERE NOT EXISTS (SELECT 1 FROM school WHERE name = '武汉大学' LIMIT 1);
```

### 3.2 `grad_employment_record` — 无 DDL 变更

| 列 / 约束 | 本期 |
|-----------|------|
| `school_id` | **保留**；新导入一律写默认校 id |
| `uk_school_student_year (school_id, student_no, graduation_year)` | **保留**；单校后同学号同届自然覆盖 |
| 业务宽表列 | **不改** |

**语义变更（文档级，非列变更）：**

- 「学校名称」Excel 列不再驱动 `school_id` 解析（见 API 设计）
- 列表查询按默认校 `school_id` 过滤；其它校历史行仍可留在表中

**可选运维（后置，非 MVP）：** 将非默认校行改挂到默认校时，须先处理唯一键冲突（同学号同届多校）：

### 3.3 `managed_report` — 唯一约束演进

**保留列（不变）：** `id`、`name`、`school_id`、`template_id`、`generate_status`、`file_path`、`fail_message`、`last_generate_time`、`deleted`、`delete_time`、`create_time`、`update_time`。

**`school_id`：** 仍 `NOT NULL`；应用写入/校正为默认校 id；**不**改为可空。

#### 3.3.1 目标唯一语义

| 规则 | 实现 |
|------|------|
| 未删记录下同一 `template_id` 仅一条 | 生成列 `active_template` + `UNIQUE` |
| 已软删 | 生成列为 `NULL`，不占唯一（可再建同模版报告） |

#### 3.3.2 与旧设计关系

| 旧（`report_manage` 库设计） | 新（本期） |
|------------------------------|------------|
| `active_school_template` = `CONCAT(school_id, ':', template_id)`（未删） | **废弃/替换**为 `active_template` = `CAST(template_id AS CHAR)` 或等价（未删） |
| `uk_active_school_template` | **替换**为 `uk_active_template` |

若现网**从未**落地生成列（仅应用层 `existsBySchoolIdAndTemplateId…`）：直接加 `active_template` 即可，无需先 drop 旧列。

若现网**已有** `active_school_template`：按 §7.2 迁移步骤 drop 旧生成列/索引后再加新列。

#### 3.3.3 生成列定义（推荐）

```text
active_template VARCHAR(32)
  GENERATED ALWAYS AS (IF(deleted = 0, CAST(template_id AS CHAR), NULL)) STORED
UNIQUE KEY uk_active_template (active_template)
```

说明：用 `CAST(template_id AS CHAR)` 而非裸 `template_id` 作生成列类型，便于与旧 `VARCHAR` 生成列迁移习惯一致；亦可使用数值型生成列（MySQL 8 支持），二选一即可，**编码前定一种**。

#### 3.3.4 加唯一前数据检查

同一 `template_id` 下若已有多条 `deleted=0`，须先软删/合并多余行，否则 `uk_active_template` 创建失败：

```sql
SELECT template_id, COUNT(*) AS cnt
FROM managed_report
WHERE deleted = 0
GROUP BY template_id
HAVING cnt > 1;
```

## 4. 业务校验与库约束分工

| 规则 | 库 | 应用 |
|------|----|------|
| 「武汉大学」名称唯一 | `uk_school_name` | `getOrCreate` + 冲突重查 |
| 报告同模版一份未删 | `uk_active_template` | `existsByTemplateIdAndDeleted*`（双保险） |
| 新就业/报告的 `school_id` = 默认校 | 无 CHECK（避免写死 id） | `DefaultSchoolService.requireId()` |
| 创建报告必填 | — | `name` + `template_id`（不再校验收前端 `schoolId`） |
| 生成中改模版 | — | 拒绝（沿用） |
| 无物理 FK | — | 写前 ensure 默认校存在；模版未删校验 |

## 7. DDL（参考）

> `ddl-auto=update` **不可靠**创建/替换 `GENERATED ... STORED`；正式环境用手跑脚本。下列按「增量变更」书写。

### 7.1 `school.name` 唯一 + 种子

```sql
-- 1) 查重
SELECT name, COUNT(*) AS cnt FROM school GROUP BY name HAVING cnt > 1;

-- 2) 去重后加唯一（按实际清理后再执行）
ALTER TABLE school
  ADD UNIQUE KEY uk_school_name (name);

-- 3) 种子默认校
INSERT INTO school (name, create_time, update_time)
SELECT '武汉大学', NOW(), NOW()
FROM DUAL
WHERE NOT EXISTS (SELECT 1 FROM school WHERE name = '武汉大学' LIMIT 1);
```

### 7.2 `managed_report`：模版唯一（无旧生成列时）

```sql
-- 查同模版多份未删
SELECT template_id, COUNT(*) AS cnt
FROM managed_report
WHERE deleted = 0
GROUP BY template_id
HAVING cnt > 1;

ALTER TABLE managed_report
  ADD COLUMN active_template VARCHAR(32)
    GENERATED ALWAYS AS (IF(deleted = 0, CAST(template_id AS CHAR), NULL)) STORED
    COMMENT '未删时模版唯一键；已删为NULL' AFTER update_time,
  ADD UNIQUE KEY uk_active_template (active_template);
```

### 7.3 `managed_report`：已有 `active_school_template` 时

```sql
ALTER TABLE managed_report
  DROP INDEX uk_active_school_template,
  DROP COLUMN active_school_template;

-- 然后执行 §7.2 的 ADD COLUMN / ADD UNIQUE
```

### 7.4 回滚思路（仅约束）

```sql
ALTER TABLE managed_report DROP INDEX uk_active_template, DROP COLUMN active_template;
ALTER TABLE school DROP INDEX uk_school_name;
-- 不自动删除「武汉大学」种子行
```

回滚后应用须同步恢复旧唯一校验逻辑，否则仅应用层约束。

## 8. 数据迁移策略（产品对齐）

| 数据 | MVP | 后置可选 |
|------|-----|----------|
| 缺少「武汉大学」行 | 应用 `getOrCreate` + §7.1 种子 | — |
| 多余其它学校行 | **保留** | 确认无引用后删除 |
| 就业行 `school_id` ≠ 默认校 | **保留**；list 不可见 | 人工/脚本改挂（处理唯一冲突） |
| 报告 `school_id` ≠ 默认校 | **保留**；update/generate 时应用校正 | 一次性 `UPDATE` 为默认校 id |
| 同模版多份未删报告 | **必须**在加 `uk_active_template` 前处理 | — |

**不推荐**在应用启动时自动全表 `UPDATE` 就业数据（风险：唯一键冲突、误覆盖）。

## 9. 风险

| 风险 | 可能性 | 影响 | 缓解 |
|------|--------|------|------|
| 重复校名导致加 `uk_school_name` 失败 | 中 | 约束落不了 | §7.1 先查重清理 |
| 同模版多报告导致加 `uk_active_template` 失败 | 中 | 约束落不了 | §3.3.4 先清理 |
| JPA 误映射生成列 | 低 | 启动/更新异常 | 实体不映射该列 |
| 仅改应用、不跑 DDL | 中 | 并发下可能插出同模版两份 | 编码验收要求执行 §7；应用层双检 |
| 历史就业未迁移 | 低（预期） | 旧校数据「消失」于 UI | 文档说明；后置脚本 |
