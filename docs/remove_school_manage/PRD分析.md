# PRD 分析：去掉学校管理（默认武汉大学）

**需求来源：** 一段话 —「去掉学校管理，现在的数据默认都是武汉大学」  
**功能短名：** `remove_school_manage`  
**状态：** DRAFT — 仅需求  
**关联现状：**

- 管理台侧边栏有 **学校管理**（`#school`）：列表 + Excel 导入；另有 `/school/import.html`
- HTTP：`GET /school/list`、`POST /school/importExcel`、`GET /school/importProgress`、`GET /school/import`
- 表 `school`；就业宽表 `grad_employment_record.school_id`；报告实例 `managed_report.school_id`
- 数据管理：按学校筛选 / 导入时指定学校；Excel「学校名称」解析为 `school_id`
- 报告管理：创建/编辑必选学校；唯一约束「同一学校 + 同一模版」仅一份未删报告；生成时按学校取就业数据
- AI 简化报告：不依赖学校选择器（与本需求弱相关）

---

## 1. 问题陈述

当前系统按「多学校」设计：需先维护学校主数据，再在就业导入、报告创建等处反复选择学校。产品侧已明确：**业务只服务武汉大学一校**，学校管理成为多余步骤，增加配置成本与选错风险。

若不收敛：运营仍要维护学校列表、在多处选校；与「数据默认都是武汉大学」的产品定位不一致。

---

## 2. 目标用户 / JTBD

**Primary：** 管理台配置 / 运营人员（使用数据管理、报告管理的同一批人）。

**JTBD：**

| 场景 | When | I want | so that |
|------|------|--------|---------|
| 进管理台 | 打开后台时 | 不再看到「学校管理」入口与流程 | 少一步无意义维护 |
| 导入就业 | 上传毕业生去向 Excel 时 | 不必选学校、也不依赖学校主数据导入 | 数据直接归属武汉大学 |
| 建报告 | 创建/编辑报告时 | 不必再选学校 | 只关心名称与模版，默认对本校数据生成 |

**Not for：** 多校 SaaS / 跨校合并报告；外部系统继续依赖「学校 CRUD」的长期多租户对接（本期按单校收敛）。

---

## 3. 成功标准

| 指标 | 目标 | 如何衡量 |
|------|------|----------|
| 学校管理不可见 | Must | 管理台无「学校管理」页签；独立学校导入页不再作为产品入口（或统一引导至数据/报告流程） |
| 默认本校 | Must | 就业导入、列表取数、报告生成所依赖的学校维度**固定为武汉大学**，用户无需选择 |
| 报告可不选校 | Must | 创建/编辑报告表单无学校下拉；仍能按武汉大学就业数据一键生成 |
| 旧多校入口收敛 | Must | 对外不再提供「维护多所学校」的产品能力（列表/导入学校）；调用方文档同步说明 |
| 历史数据可读 | Should | 库内若仍有非武大 `school_id` 行：策略明确（忽略 / 迁移 / 清理），且不阻断武大主流程 |

---

## 4. 方案概述（What）

```text
去掉学校管理（单校默认：武汉大学）
├── 产品面
│   ├── 移除管理台「学校管理」及学校 Excel 导入入口
│   ├── 数据管理：去掉「按学校筛选 / 导入指定学校」
│   └── 报告管理：去掉学校选择；绑定固定本校
├── 数据语义
│   └── 就业数据、报告实例的学校维度默认 = 武汉大学
└── 对外契约
    └── 学校列表/导入类 API 下线或废弃；报告 create 不再要求调用方传 schoolId
```

**边界：** 本需求负责「去掉学校管理产品能力 + 全链路默认武汉大学」；**不重做**就业宽表字段、**不重做**模版/指标配置、**不重做** Word 引擎。是否物理删除 `school` 表或仅保留一条「武汉大学」记录，属实现策略（技术方案再定），产品上用户不可再管多校。

## 5. 功能需求

### 5.1 管理台 / 入口（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| U-1 | 侧边栏移除「学校管理」 | Must | `#school` 分区下线 |
| U-2 | 首页/引导文案去掉「先导入学校」步骤 | Must | 现有「学校 → 数据 → 模版 → 报告」改为从数据/模版起 |
| U-3 | `/school/import`、独立学校导入页不再作为可用产品入口 | Must | 可 404 / 重定向到数据管理 — 见 Q1 |
| U-4 | 数据管理：去掉学校筛选下拉、导入时「指定学校」下拉 | Must | 列表默认即本校（或全量即本校） |
| U-5 | 报告管理：创建/编辑去掉学校下拉与列表「学校」列（或固定展示「武汉大学」） | Must | 展示策略见 Q2 |

### 5.2 默认学校语义（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| D-1 | 系统内默认学校名称为 **「武汉大学」**（精确匹配用此中文名） | Must | ID 是否固定见 Q3 |
| D-2 | 就业 Excel 导入：不再要求用户选学校；落库一律归属默认学校 | Must | Excel 中「学校名称/学校ID」列策略见 Q4 |
| D-3 | 报告生成取数：始终按默认学校过滤就业数据 | Must | 与现「按 schoolId 聚合」语义一致，仅学校固定 |
| D-4 | 报告创建/更新：调用方**不再必填** `schoolId`；服务端写入默认学校 | Must | 若仍传 `schoolId` 见 Q5 |
| D-5 | 首次使用或空库时，系统须能确保默认「武汉大学」可用（自动创建或初始化） | Must | 避免因无学校记录导致导入/建报告失败 |

### 5.3 业务规则调整（Must / Should）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| B-1 | 报告唯一性：由「学校 + 模版」改为「模版」维度唯一（同模版仅一份未删报告），或保留库字段但学校恒为默认校 | Must | 产品拍板见 Q6 |
| B-2 | 学校列表 / 学校 Excel 导入能力对产品用户关闭 | Must | API 废弃策略见 Q7 |
| B-3 | 库内非「武汉大学」的历史就业/报告数据 | Should | 迁移到武大 / 隐藏 / 保留但不在 UI 展示 — 见 Q8 |

### 5.4 明确不做（Won't / 后置）

| 项 | 原因 |
|----|------|
| 多学校切换、学校权限、跨校合并 | 与「默认都是武汉大学」相反 |
| 改就业表业务列结构（除学校归属语义） | 未要求 |
| 改 AI 简化报告主流程 | 本不依赖学校选择器 |
| 本期强制物理删表 `school` | 可为实现细节；产品上不可见即可 |

## 6. 关键用户流程（变更后）

1. 管理台 → **数据管理** → 直接导入就业 Excel（无需先建学校、无需选校）  
2. **模版管理** → 配置指标（不变）  
3. **报告管理** → 填报告名称、选模版 → 保存（无选校）  
4. 点 **生成** → 系统按 **武汉大学** 就业数据聚合 → 出 Word  

---

## 7. MVP 范围

**MVP：**

- 管理台去掉学校管理及相关选校 UI  
- 就业导入 / 报告创建·生成默认武汉大学  
- 报告不再要求用户选择学校  
- 学校导入/列表不再作为产品能力（文档标明废弃）  

**可后置：**

- 历史多校数据一键清洗工具  
- 物理删除 `school` 表与全部 `/school/*` 代码  
- 配置项可改「默认学校名」（本期写死武汉大学即可，见 Q9）

## 9. 待确认

- [重定向到数据管理] **Q1** 旧 `/school/import`、静态导入页：直接下线（404），还是重定向到 `#data`？  
- [列表固定展示「武汉大学」] **Q2** 报告列表：完全去掉「学校」列，还是固定显示「武汉大学」文案？  
- [按名称确保存在即可，不强制固定 id] **Q3** 默认校：是否要求固定主键 id（如恒为 1），还是「按名称武汉大学查找/自动创建」即可？  
- [忽略 Excel 学校列，强制归属武大] **Q4** 就业 Excel 仍含「学校名称/学校ID」时：忽略并强制武大，还是与表头不一致则报错？  
- [忽略调用方 schoolId，强制默认校] **Q5** 报告 create/update 或 `createReport` 若仍传 `schoolId`/`lSchoolId`：忽略强制默认，还是校验必须等于武大否则失败？  
- [同模版仅一份未删报告] **Q6** 唯一性：改为「同一模版仅一份」，还是「仍按学校+模版」但学校恒为武大（效果等价于一模版一份）？  
- [API 直接下线] **Q7** `/school/list`、`/school/importExcel` 等：直接返回失败/404，还是保留内部只读一段时间？  
- [历史非武大数据：导入后可视作需人工清理；MVP 列表可只展示武大] **Q8** 已有多校数据如何处理？迁移全部 `school_id`→武大 / 仅 UI 过滤武大 / 其它？  
- [写死「武汉大学」] **Q9** 默认校名是否允许配置（环境变量），还是代码/文档写死？  
- [同步改调用方文档] **Q10** `docs/admin_console.md`、`school_excel_import.md`、`report_manage.md` 等是否在编码同次交付中标注废弃并改示例？（接口变更规则要求：契约变则须更新 `docs/`）

## 10. 风险

| 风险 | 可能性 | 影响 | 缓解 |
|------|--------|------|------|
| 库中无「武汉大学」记录导致导入/建报告失败 | 中 | 主流程不可用 | D-5 自动确保存在 |
| 历史多校数据混在列表中造成困惑 | 中 | 运营误读 | Q8 明确过滤或迁移 |
| 外部脚本仍调 `/school/*` 或传 `schoolId` | 中 | 对接中断 | Q5/Q7 + 调用方文档 |
| 唯一约束从「校+模版」调整引发重复报告 | 低 | 建报告失败或脏数据 | Q6 与迁移策略一起定 |
| 就业 Excel「学校名称」与强制武大冲突 | 低 | 导入误解 | Q4 产品文案写清 |

## 11. Evidence

Assumption — 来自产品口述「去掉学校管理，数据默认都是武汉大学」；尚无多校使用量统计。建议上线后用「学校相关接口调用次数 → 0」与「就业导入/建报告成功率」验证。

---

## 12. Hypothesis

We believe **去掉学校管理并将就业/报告默认绑定武汉大学** will **减少无意义选校与学校维护** for **单校运营人员**.  
We'll know we're right when **管理台无学校管理入口，导入就业与创建报告均无需选校，且生成仍按武汉大学数据成功出稿**.
