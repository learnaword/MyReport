# 技术方案：报告管理

**需求来源：** `docs/report_manage/PRD分析.md`、`docs/report_manage/数据库设计.md`  
**功能短名：** `report_manage`  
**状态：** DRAFT — 仅方案，不编码  
**目标：** 管理台可管报告实例（名称/学校/模版），一键按模版树聚合就业数据并组装为现有 Word 引擎可消费的 `reportJsonArr`，异步生成 `.docx`。

---

## 1. 目标与非目标

### 1.1 目标

| # | 能力 |
|---|------|
| 1 | 报告实例 CRUD（软删）+ 同校同模版唯一 |
| 2 | 生成：读模版树 → 按校聚合指标 → 组装 JSON → 调现有 `SpireReportUtil.createReport` |
| 3 | 库表同步生成状态 / 成功路径；Redis 仍负责任务进度与锁 |
| 4 | 管理台「报告管理」面板 |

### 1.2 非目标

- 不重写 Spire/POI 渲染；不改模版树编辑
- 不建指标结果表、不预聚合缓存（MVP）
- 不替换外部直接调用的 `POST /report/createReport`（可并存）
- 不做完整下载中心 UI（路径落库 + Redis，Q14）

---

## 2. 现状与切入点

| 能力 | 现状 | 本方案用法 |
|------|------|------------|
| Word 生成 | `SpireReportUtil.createReport` 后台线程；Redis `REPORT_LIST` / `REPORT_LOCK` | 组装完后**内聚调用**，`reportId = managed_report.id` |
| 静态指标 | `nType=0`：节点带 `strData`（JSON 字符串）+ `strShowList`；`getStaticMetricsData` 解析后渲染 | METRIC 一律走**静态指标**路径，不走客观题数据集 |
| 表格/图 | `strShowList[].strType`：`1` 表 / `2` 图 | `TABLE`→1，`CHART`→2 |
| 模版 | `ReportTemplateService` detail 树 | 生成时只读 |
| 就业数据 | `grad_employment_record` | 按 `school_id` 加载后内存分组（MVP）；不过滤届次 |
| 管理台 | `/admin/index.html` 多 panel | 新增 `report` panel |

**引擎树节点关键字段（已从代码确认，补齐 PRD Q11）：**

| 模版节点 | 生成树节点要点 |
|----------|----------------|
| TITLE | `nTreeType=0`，`strTitle=name`，`strContent=intro`（可空），`children` |
| METRIC | `nTreeType=1`，`nType=0`，`strTitle=指标名`，`strData=指标结果 JSON 字符串`，`strShowList=[{ strType, bShow, bPreant… }]`，通常无 `children` |

指标结果 JSON 形态见 PRD §6（`title` / `headList` / `tableList` / …）。

## 3. 总体架构

```text
管理台 / 外部 HTTP
        │
        ▼
ManagedReportController   （/managed-report/*）
        │
        ▼
ManagedReportService
  ├── CRUD（JPA ManagedReport）
  ├── generate(reportId)
  │     ├── 校验：未删、非 GENERATING、模版启用、学校存在
  │     ├── ReportTemplateService → 模版树 + 外观
  │     ├── MetricAggregateService → 按校 + statField 聚合
  │     ├── ReportJsonAssembler → reportJsonArr + overallSetting
  │     ├── 置 GENERATING，记 last_generate_time
  │     ├── SpireReportUtil.createReport(...)
  │     └── 注册/触发 终态回写（成功 file_path / 失败 fail_message）
  └── delete：软删 + 尽力删磁盘文件
```

分层与现有一致：`controller` → `service` → `entity`/`repository`；Word 仍在 `util.word`；**不**新建平行 Redis 客户端。

## 4. 模块设计

### 4.1 持久化

| 类 | 职责 |
|----|------|
| `ManagedReport` | 映射 `managed_report`（见数据库设计） |
| `ManagedReportRepository` | 分页列表、按 id 查未删、唯一冲突依赖 DB 生成列 |

生成列 `active_school_template`：优先用原生 DDL 初始化；JPA 实体可**不映射**该列（只读库维护），避免 `ddl-auto` 与 STORED 生成列冲突。

### 4.2 ManagedReportService（编排）

**创建 / 更新**

- 校验 name、school、template（未删；创建时建议 status=1）
- 捕获唯一约束冲突 → 友好「该学校已存在绑定此模版的报告」
- `GENERATING` 时拒绝改 `school_id` / `template_id`（库设计 D2 默认）

**生成 `generate(id)`**

1. 加载实例；若 `generate_status==GENERATING` → 拒绝  
2. 校验模版 `deleted=0 && status=1`（Q15）  
3. 组装 JSON（失败则直接 `FAILED` + message，不调 Spire）  
4. 写库 `GENERATING`，清空本次 `fail_message`（保留旧 `file_path` 至成功覆盖，Q3）  
5. 构造 `CreateReportVO`：`reportId=id.intValue()`，`reportName=name`，`lSchoolId=schoolId`  
6. `overallSetting`：至少  
   - `strCoverImg` ← 模版 `coverImage`  
   - `strBackCoverImg` ← 模版 `backCoverImage`  
   - `nMetricsCount` ← 本次实际 METRIC 数（进度用；无则给保守默认）  
   - `dataSetConfigList` ← 空数组即可（静态指标会 `datasetMap.put(0, …)`）  
7. 调用 `SpireReportUtil.createReport`  
8. **终态回写**见 §6  

**删除**

- 软删；尽力 `Files.delete` 旧 `file_path`；置 `file_path=null`

### 4.3 MetricAggregateService（指标计算）

**输入：** `schoolId` + 模版树上全部 METRIC 的 `statField` / `valueFormat`  
**输出：** `Map<nodeId, JSONObject>` 指标结果（PRD 形态）

**算法（单校 MVP）：**

1. `findBySchoolId(schoolId)` 加载该校就业行（不过滤届次，Q6）  
2. 对每个 METRIC：  
   - `statField` → 白名单映射到实体 getter / 列（非法则**跳过该节点**，Q10，记 warn）  
   - 遍历行：非空值计入分组 `Map<String, Long>`  
   - 分母 `nonNullTotal` = 该字段非空行数（Q7）  
   - `COUNT`：组计数 → 字符串整数  
   - `PERCENT`：`count * 100.0 / nonNullTotal`，`DecimalFormat` **两位小数 + `%`**（Q8），分母 0 则跳过  
3. 组装：

```json
{
  "strText": "<最近祖先 TITLE 的 intro，否则 \"\">",
  "strText2": "",
  "headList": [["名称层", "数据"]],
  "chartDataList": [],
  "tableList": [["分组A", "12"], ...],
  "cellHeadList": [],
  "title": "<metric.name>",
  "cellTableList": []
}
```

**`statField` 白名单：** 与 `grad_employment_record` 业务列对齐（如 `college_name`、`destination_category`…），配置侧存 **库列名 / 约定别名** 须与白名单一致；建议与模版配置文档列一张对照表（API 设计可附）。禁止反射任意字段名，防注入式列访问。

**性能：** 单校全量进内存可接受则 MVP 不做 SQL GROUP BY；若单校十万级行再改为「按字段一次 GROUP BY」或批量查询。

**CHART：** MVP 与 TABLE 共用 `tableList`；`chartDataList` 可先 `[]`，`strShowList.strType=2` 时依赖引擎从同一 `strData` 取数（若图表缺字段导致空白，记为后置对齐，里程碑 4）。

### 4.4 ReportJsonAssembler（模版树 → reportJsonArr）

递归模版 `nodes`：

```text
function mapNode(templateNode, parentIntro):
  if TITLE:
    return {
      nTreeType: 0,
      strTitle: name,
      strContent: intro or "",
      strNote: "",
      children: map children（传入 intro 供子 METRIC 的 strText）
    }
  if METRIC:
    data = aggregateResult.get(id)
    if data == null: return null  // 跳过
    showType = displayType==CHART ? 2 : 1
    return {
      nTreeType: 1,
      nType: 0,
      strTitle: name,
      strData: data.toJSONString(),  // 含 title/headList/tableList；strText 用 parentIntro
      strShowList: [{ strType: showType, bShow: true, bPreant: true }],
      children: []
    }
```

根数组过滤 `null`；若过滤后为空 → 生成失败「无可用指标数据」。

### 4.5 管理台

- 侧栏增加「报告管理」→ `data-panel="report"`，hash `#report`  
- 表单：名称、学校下拉（复用 `/school/list`）、模版下拉（启用模版 `/report-template/list?status=1`）  
- 列表：名称、学校名、模版名、状态、时间、操作（生成 / 编辑 / 删除）  
- 生成：调 `POST .../generate`，前端可短轮询详情或 Redis 进度（与导入进度类似）；MVP 可仅刷新列表看 `generate_status`

样式与现有 admin 面板一致，不大改视觉体系。

## 5. 关键时序

```text
用户点「生成」
  → Controller.generate
  → Service：校验 + 聚合 + 组装
  → DB: generate_status=GENERATING
  → SpireReportUtil.createReport（新线程出 Word）
  → HTTP 立即返回 code=0（任务已提交）
  →（异步）Word 成功/失败
  → 终态回写 DB：SUCCESS+file_path / FAILED+fail_message
  → Redis 进度终态 2 / 4（既有）
```

与直接调 `/report/createReport` 的差异：调用方**不必**自备 `reportJsonArr`；由本服务根据模版与就业数据生成。

## 6. 生成终态回写（关键设计点）

现状：`SpireReportUtil` 只写 Redis，**不写业务表**。

**推荐方案（最小侵入）：在 `overallSetting` 或 `CreateReportVO.attributes` 中带标记 `managedReportSync=true`，于 `createSingleReport` 的 success / catch 末尾调用同步钩子。**

```text
成功：state=2 → ManagedReportGenerateSync.onSuccess(reportId, finalFilePath)
失败：state=4 → onFailure(reportId, message)
```

实现要点：

- 钩子通过 **Spring 可注入的单例** 暴露静态入口（与现有 `RedisTemplate` 静态用法一致），或 `ApplicationContext` 懒取 Bean，避免循环依赖。  
- `onSuccess`：`generate_status=SUCCESS`，写 `file_path`（若路径变更则尽力删旧文件，D3 建议），清空 `fail_message`。  
- `onFailure`：`generate_status=FAILED`，写可读 `fail_message`（截断 512）。  
- 进程宕机导致永久 `GENERATING`：启动时可扫超时任务置 FAILED（可选，后置；或复用类似 `ImportProgressStartupCleaner` 思路）。

**备选（不改 Spire）：** 提交后起轮询线程读 Redis `REPORT_LIST` 直至终态再写库——实现简单但多一次轮询组件，锁与超时需自管。优先钩子方案。

**锁：** 继续用 `REPORT_LOCK:{reportId}`；生成入口若发现 Redis 仍持锁或库状态 GENERATING 则拒绝。是否在入口主动 `setnx` 锁：若 Spire 未统一加锁，可在 Service 入口补 `set`+TTL，与删除锁的现有 catch 路径对齐（实现时核对 Spire 是否已 set 锁；当前代码多见 delete，**入口补锁**更稳）。

## 9. 与 PRD / 库表对齐摘要

| 项 | 方案落地 |
|----|----------|
| 持久化 + 唯一校模版 | `managed_report` + 生成列唯一 |
| reportId | 表 `id` |
| 不过滤届次 | 查询仅 `school_id` |

| strText | 父 TITLE `intro` |
| headList 固定 | `名称层` / `数据` |
| 模版停用禁生成 | generate 前校验 status |

## 10. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 静态指标 `strShowList` 缺字段导致不渲染 | 对照一次真实成功样例 JSON 固化最小 `strShowList`；编码前用样例单测 Assembler 输出 |
| CHART 与 `chartDataList` 协议不全 | MVP 以 TABLE 为主；CHART 后置验证 |
| `statField` 与列名不一致 | 白名单 + 模版配置校验提示 |
| Spire 静态方法难测 | Aggregator / Assembler 单测；生成联调 `make run` + 小数据学校 |
| GENERATING 僵尸状态 | 启动清理或超时 FAILED |
| 改配后旧 file_path | UI 展示「上次成功文件，改配后请重生成」 |
