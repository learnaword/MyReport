# PRD 分析：报告管理

**需求来源：** 一段话 —  
1）报告管理：创建时配置报告名称、学校、模版；  
2）点击生成：获取模版配置的 JSON 结构，再按 JSON 生成报告；  
3）指标配置：按配置字段计算每个元素的数量或百分比，结果组装为生成侧节点 JSON（含 `strText` / `headList` / `tableList` 等）。

**功能短名：** `report_manage`  
**状态：** DRAFT — 仅需求  
**关联能力：**  
- 模版结构与指标元数据：`report_template_config`（已有）  
- 学校主数据 / 就业去向数据：`school`、`grad_employment`（已有导入）  
- Word 生成入口：现有 `POST /report/createReport`（消费 `reportJsonArr`）

---

## 1. 问题陈述

当前报告生成依赖**调用方自行拼好** `reportJsonArr` 再调 `createReport`。模版侧虽已能配置标题树与指标元数据（字段、数量/百分比、图表/表格），但缺少：

1. **报告实例管理**：按「名称 + 学校 + 模版」落库、列表、再生成；  
2. **从模版到生成 JSON 的组装**：根据模版树拉出结构，对每个 METRIC 用配置字段做统计，填成生成引擎认识的节点数据；  
3. **指标计算结果的标准形态**：与现有 Word 管线字段对齐（见 §6）。

若不做：模版配置与报告生成仍断裂，运营需手工或外部系统拼 JSON，易错且难复用。

## 2. 目标用户 / JTBD

**Primary：** 管理台配置 / 运营人员（与模版管理同一入口体系）。

**JTBD：**

| 场景 | When | I want | so that |
|------|------|--------|---------|
| 建报告 | 要为一所学校出一份报告时 | 选名称、学校、模版并保存 | 有可管理的报告实例，不必直接调生成 API |
| 生成 | 报告与数据就绪时 | 一点「生成」即按模版出 Word | 不用手工拼 `reportJsonArr` |
| 指标 | 模版已配好 `statField` / 数量或百分比时 | 系统按学校数据算好每个指标块 | 表格/图表节点有真实数据 |

**Not for（本期默认）：** 外部业务系统大批量 API 编排（可后置开放只读/生成接口）；学生端自助下载。

## 3. 成功标准

| 指标 | 目标 | 如何衡量 |
|------|------|----------|
| 可创建报告实例 | Must | 填写名称 + 学校 + 模版后可持久化并列表可见 |
| 生成可走通 | Must | 「生成」后能拉到模版树，组装 `reportJsonArr`，并触发既有 Word 异步生成 |
| 指标可算 | Must | METRIC 按 `statField` + `valueFormat`（COUNT/PERCENT）产出节点 JSON；至少填充示例形态中的 `title`、`headList`、`tableList` 等约定字段 |
| 进度可感 | Should | 与现有报告进度/锁约定一致，可轮询或管理台展示状态 |
| 失败可读 | Must | 无模版、无学校数据、字段缺失等有明确失败原因 |

## 4. 方案概述（What）

```text
报告管理
├── 创建 / 编辑：报告名称、学校、模版
├── 列表：查看状态、再次生成（待确认）
└── 生成
    ├── 读模版 detail（标题 + 指标树 + 外观）
    ├── 按学校取数
    ├── 对每个 METRIC：按配置字段聚合 → 节点数据块
    ├── 组装 reportJsonArr（+ overallSetting）
    └── 调用既有报告生成能力 → 落盘 .docx + Redis 进度
```

**边界：** 本功能负责「报告实例 + 取模版 + 指标计算 + 组装 + 触发生成」；**不重做** Word 渲染引擎；**不重做**模版树编辑（沿用模版管理）。

## 5. 功能需求

### 5.1 报告管理（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| R-1 | 创建报告：必填 **报告名称、学校、模版** | Must | 模版须启用且未删；学校须存在 |
| R-2 | 报告列表：至少展示名称、学校、模版、创建/更新时间、生成状态 | Must | 分页 |
| R-3 | 可编辑未生成或允许改配后的名称/学校/模版 | Should | 改配规则见待确认 Q3 |
| R-4 | 删除报告实例 | Should | 软删优先；是否删已生成文件见 Q4 |
| R-5 | 管理台入口：「报告管理」 | Must | 与现有 `/admin` 侧边栏并列 |

### 5.2 点击生成（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| G-1 | 从报告绑定的模版读取配置 JSON（树结构） | Must | 等价于现有 `detail` 树 |
| G-2 | 将模版 TITLE 映射为报告章节节点；METRIC 映射为带数据的内容节点 | Must | 映射细则见 §6 |
| G-3 | 组装完成后触发 Word 生成（异步） | Must | 对齐现有 `createReport` 语义：提交成功 ≠ 文件已就绪 |
| G-4 | 生成中禁止重复提交或需排队/拒绝 | Must | 与现有 `REPORT_LOCK` 或新锁约定对齐（见 Q5） |
| G-5 | 外观：模版封面/底图映射进全局配置 | Should | 若已有字段：`coverImage`→封面、`backCoverImage`→底图 |
| G-6 | 生成失败写明原因，报告状态可回到可重试 | Must | |

### 5.3 指标配置 → 计算结果（Must）

模版节点已具备（来自 `report_template_config`）：

- `statField`：数据集字段名  
- `valueFormat`：`COUNT` / `PERCENT`  
- `displayType`：`TABLE` / `CHART`  
- `chartStyle`：`BAR` / `PIE` / `LINE`（图表时）  
- `name`：指标标题  

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| M-1 | 按 `statField` 从**该报告所选学校**的数据集中取字段值做分组聚合 | Must | 数据集默认见 Q6（建议就业去向宽表） |
| M-2 | `COUNT`：各组计数（或约定分子）写入结果列 | Must | |
| M-3 | `PERCENT`：各组占比（分母约定见 Q7）写入结果列 | Must | 小数位数见 Q8 |
| M-4 | 结果组装为生成侧节点对象（字段见 §6） | Must | 用户给定形态为基准 |
| M-5 | `displayType=TABLE` 时优先填充 `headList` + `tableList` | Must | |
| M-6 | `displayType=CHART` 时填充 `chartDataList`（及/或与表格同源数据） | Should | 与现有引擎字段对齐，缺约定标待确认 |
| M-7 | `strText` / `strText2`：可由标题 intro、固定话术或计算摘要生成 | Could | 本期可空串，见 Q9 |
| M-8 | 字段不存在或全空：该指标失败策略 | Must | 跳过 / 整单失败 — 见 Q10 |

### 5.4 明确不做（Won't / 后置）

| 项 | 原因 |
|----|------|
| 在报告管理里编辑模版树 | 已有模版管理 |
| 在线改 Word 排版 / 新图表类型 | 未要求 |
| 跨学校合并报告 | 未要求 |
| 指标公式 DSL / 自定义 SQL | 仅「按字段计数或百分比」 |
| 替换 Spire 生成引擎 | 复用现有 |

## 6. 指标结果 JSON 形态（生成侧约定）

用户给定的目标形态（每个 METRIC 对应一块，或嵌在章节树中 — 树挂载方式见 Q11）：

```json
{
  "strText": "",
  "strText2": "",
  "headList": [
    ["名称层", "数据"]
  ],
  "chartDataList": [],
  "tableList": [
    ["公路工程学院", "3.77"],
    ["轨道技术学院", "4.07"],
    ["建筑与测绘工程学院", "3.73"],
    ["交通信息学院", "4.52"],
    ["经济管理学院", "3.87"],
    ["汽车运用工程学院", "3.92"],
    ["铁道工程学院", "3.80"],
    ["铁道运输学院", "4.00"],
    ["智能汽车制造学院", "3.70"]
  ],
  "cellHeadList": [],
  "title": "指标标题",
  "cellTableList": []
}
```

**字段语义（产品约定）：**

| 字段 | 含义 | MVP |
|------|------|-----|
| `title` | 指标名称（来自节点 `name`） | Must 填充 |
| `headList` | 表头二维数组 | Must（TABLE）；示例默认 `["名称层","数据"]`，列名是否可配见 Q12 |
| `tableList` | 表体：每行 `[分组名, 数值字符串]` | Must |
| `chartDataList` | 图表数据 | CHART 时 Should；结构对齐现有引擎 |
| `strText` / `strText2` | 正文说明 | Could，默认可 `""` |
| `cellHeadList` / `cellTableList` | 合并单元格等 | Could，默认可 `[]` |

**与 `valueFormat` 的关系（产品理解）：**

- `COUNT` → `tableList` 第二列为整数计数字符串（如 `"12"`）  
- `PERCENT` → 第二列为百分比或小数（示例为 `"3.77"` 类；**单位是「%」还是「小数」待确认 Q8**）  
- 分组维度：对 `statField` 取值分组；示例行为「学院」——是否固定学院维、还是「字段值即分组」见 **Q13**

## 7. 关键用户流程

1. 管理台 → **报告管理** → 新建  
2. 填写 **报告名称**，选择 **学校**、**模版** → 保存  
3. 列表中对该报告点 **生成**  
4. 系统：加载模版树 → 按学校取数 → 逐指标聚合 → 组装 `reportJsonArr`（及全局设置）→ 提交异步生成  
5. 用户等待进度结束 → 下载 / 打开 `.docx`（下载入口见 Q14）

## 8. MVP 范围

**MVP：**

- 报告 CRUD（至少创建 + 列表；编辑/删除可减配到「仅创建+列表」若时间紧 — 默认含软删）  
- 绑定名称 / 学校 / 模版  
- 一键生成：读模版 JSON + 指标 COUNT/PERCENT 聚合 + 组装节点 + 触发现有 Word 生成  
- TABLE 型指标：`title` + `headList` + `tableList` 必填正确  

**可后置：**

- 精美进度条 UI、历史版本对比  
- CHART 全样式对齐与预览  
- `strText` 自动话术  
- 外部系统开放 API  
- 多数据源 / 届次过滤 UI

## 10. 待确认

- [持久化] **Q1** 报告实例是否必须持久化（推荐：是），还是「仅临时生成、不落库」？  
- [不允许] **Q2** 同一学校 + 同一模版是否允许多份报告？唯一性规则？  
- [允许修改，修改后不清空] **Q3** 生成中 / 已成功后，是否允许改学校或模版？改后是否清空旧文件？  
- [删除] **Q4** 删除报告时，已生成的 `.docx` 是否同步删除？  
- [沿用reportId] **Q5** 防重入：沿用现有 `reportId` 锁，还是按「报告实例 ID」新 key？`reportId` 如何分配？  
- [不过滤] **Q6** 指标数据源：默认仅就业去向表？是否按 `graduationYear` 过滤？届次是否报告级可选？  
- [全校该字段非空人数] **Q7** PERCENT 分母：全校该字段非空人数 / 全校总人数 / 其它？  
- [2位小数，带%号] **Q8** PERCENT 展示：保留几位小数？是否带 `%` 符号？示例 `3.77` 的单位含义？  
- [用 TITLE `intro`] **Q9** `strText`/`strText2`：固定空、用 TITLE `intro`、还是模板句？  
- [跳过该节点并继续] **Q10** 单指标算失败：跳过该节点并继续，还是整单失败？  
- [需对照现有成功样例 JSON 定稿] **Q11** `reportJsonArr` 树形：TITLE 与 METRIC 如何嵌套进现有引擎节点协议（字段名、`type` 等）？需对照现有成功样例 JSON 定稿。  
- [固定] **Q12** `headList` 列名是否可配，还是固定「名称层 / 数据」？  
- [始终按 `statField` 取值分组即可] **Q13** 分组维度：始终按 `statField` 取值分组即可，还是另需「维度字段」（如学院）与「度量字段」分离？  
- [仅落盘路径写入 Redis/库] **Q14** 生成完成后下载：管理台提供下载链接，还是仅落盘路径写入 Redis/库？  
- [不允许] **Q15** 模版停用后，已绑定该模版的报告是否仍允许生成？

## 11. 风险

| 风险 | 可能性 | 影响 | 缓解 |
|------|--------|------|------|
| 生成侧节点协议与模版树不完全同构 | 高 | 组装出的 JSON 无法渲染 | Q11 用真实样例对齐；MVP 先支持 TABLE |
| `statField` 与真实列名不一致 | 高 | 指标全空或失败 | 字段白名单；配置时校验；失败可读 |
| PERCENT 口径争议 | 中 | 业务不认可数字 | Q7/Q8 产品拍板后再编码 |
| 大数据量学校聚合慢 | 中 | 生成超时体感差 | 异步 + 进度；必要时预聚合后置 |
| 与模版外观路径约定不一致 | 低 | 封面丢失 | 复用已确认的 cover/backCover 映射 |

## 13. Hypothesis

We believe **「报告实例（名称+学校+模版）+ 一键按模版指标聚合并生成 Word」** will **消除手工拼 `reportJsonArr`、打通模版到成稿** for **管理台运营人员**.  
We'll know we're right when **选定学校与启用模版后，一键生成可产出含表格指标数据的 `.docx`，且无需调用方自备节点 JSON**.
