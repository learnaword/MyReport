# PRD 分析：指标统计字段下拉选择

**需求来源：** 一段话 —「指标配置的时候，统计字段前端设计为下拉框，可以选择，展示中文，值是数据库里字段」  
**功能短名：** `metric_field_select`  
**状态：** DRAFT — 仅需求  
**关联现状：**

- 管理台模版「编辑结构」中，METRIC 的「统计字段」为**自由文本输入**（placeholder：数据集字段名）
- 落库字段仍为 `statField`（库列名 / 约定名，如 `college_name`）
- 生成侧已有 `statField` **白名单**（与 `grad_employment_record` 业务列对齐）；非法字段会被跳过
- 中文名已有权威对照：`docs/graduate_employment/数据库设计.md` 表头 ↔ 列名

---

## 1. 问题陈述

配置人员在配指标时需手填数据库列名（如 `college_name`）。易出现拼写错误、记不住字段、或填入非白名单字段，导致生成时该指标被跳过、报告缺块。

若不改为可选列表：配置门槛高、试错成本大，且「展示中文 / 存库英文字段」的认知负担一直存在。

---

## 2. 目标用户 / JTBD

**Primary：** 管理台模版配置人员（同既有「模版管理 → 编辑结构」）。

**JTBD：**  
When 我在新增/编辑指标并选择统计维度时，I want 从下拉列表按中文名挑选字段，so I can 正确写入库列名且不必记忆 `snake_case`。

**Not for：** 报告一键生成操作员（不改模版结构）；外部直接调 `createReport` 且自带完整 `reportJsonArr` 的调用方（本需求不改其入参形态）。

---

## 3. 成功标准

| 指标 | 目标 | 如何衡量 |
|------|------|----------|
| 可选可懂 | Must | 新增/编辑指标时，「统计字段」为下拉；选项展示中文 |
| 存值正确 | Must | 选中后提交/落库的 `statField` 为对应库列名（如展示「学院名称」→ 值 `college_name`） |
| 回显正确 | Must | 编辑已有指标时，下拉按已存 `statField` 选中对应中文项 |
| 与可聚合字段一致 | Must | 下拉选项 ⊆ 生成侧可识别的统计字段集合（避免选出后生成跳过） |
| 必填仍成立 | Must | 未选统计字段时，保存被拒绝并提示（与现「名称与统计字段必填」一致） |

---

## 4. 方案概述（What）

在模版结构配置的 **METRIC 表单**中，将「统计字段」从文本框改为**单选下拉**：

```text
指标表单
├── 指标名称（文本）
├── 统计字段（下拉）← 展示中文 label，option value = 库列名
├── 输出形式 COUNT / PERCENT
├── 展示形式 TABLE / CHART
└── 图表样式（图表时）
```

**存储契约不变：** 节点仍只存 `statField` 字符串（库列名）；不新增「中文名字段」列。中文仅用于配置 UI 展示。

**选项来源（产品约束，实现选型见后续）：** 选项集合须与就业宽表可统计列及生成白名单对齐；中文文案优先对齐导入表头 / `graduate_employment` 字段映射总表。

---

## 5. 功能需求

### 5.1 下拉选择（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| S-1 | METRIC 新增表单：统计字段为下拉单选 | Must | 覆盖「添加指标」 |
| S-2 | METRIC 编辑表单：同上，并按已存值回显 | Must | 覆盖「编辑指标」 |
| S-3 | 选项 **label = 中文名**，**value = 数据库列名** | Must | 例：学院名称 / `college_name` |
| S-4 | 提交保存时 `statField` 传 value（列名），不传中文 | Must | API 契约字段名不变 |
| S-5 | 未选择时不可保存，提示必填 | Must | |
| S-6 | 选项列表与生成侧可聚合字段集合一致 | Must | 避免「能选但不能算」 |
| S-7 | 树节点摘要/列表旁注若展示 `statField`：可读性见 Q3 | Should | 可继续显示列名，或显示中文 |

### 5.2 历史数据兼容（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| C-1 | 已存合法 `statField`：编辑时下拉正确选中 | Must | |
| C-2 | 已存值不在当前选项中（历史脏数据 / 已下线字段）：见 **Q2** | Must | 策略待确认 |

### 5.3 明确不做（Won't）

| 项 | 原因 |
|----|------|
| 支持手填任意自定义字段名 | 与白名单/防误配目标冲突；见 Q1 |
| 多选统计字段 / 复合公式 | 未要求；聚合仍是单字段分组 |
| 改生成侧聚合算法或 `valueFormat` 语义 | 仅配置 UX |
| 为每个节点另存中文 `statFieldLabel` | 中文可由字典反查；避免双写不一致 |
| 非管理台前端的其它客户端改造 | 本仓管理台为当前配置入口；若另有调用方改模版 API，可选后续提供字典接口 |

## 6. 信息结构

```text
选项项（配置用，不落库）
├── value: string   // 库列名，如 college_name → 写入 node.statField
└── label: string   // 中文，如 学院名称 → 仅 UI

ReportTemplateNode（不变）
└── statField: string  // 仍存库列名
```

中文对照基准（产品）：`docs/graduate_employment/数据库设计.md` §2 表头列；与导入 Excel 表头一致处应优先同名，避免「配置叫 A、导入叫 B」。

---

## 7. 关键用户流程

1. 模版管理 → 编辑结构 → 添加/编辑指标  
2. 「统计字段」打开下拉，看到中文列表（如「学院名称」「专业名称」「毕业去向类别」…）  
3. 选择一项 → 内部值为对应列名  
4. 填写其余项并保存 → 详情/树回显；库中 `stat_field` 为列名  
5. 报告生成时仍按原 `statField` 聚合（行为不变）

---

## 8. Hypothesis

We believe **统计字段中文下拉（值=库列名）** will **降低配错字段、减少无效指标** for **模版配置人员**.  
We'll know we're right when **新配指标不再因手误列名被生成跳过，且配置人员无需查表记英文字段名**.

---

## 9. 风险

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| 中文文案与导入表头不一致 | 中 | 配置认知混乱 | 以 `graduate_employment` 映射表为单一文案源 |
| 下拉选项与生成白名单漂移 | 中 | 能选不能算或能算不能选 | 选项与白名单同源维护（实现阶段约束） |
| 历史脏 `statField` 无法回显 | 低 | 编辑时空白或错选 | Q2 明确展示/纠正策略 |
| 字段过多导致难找 | 中 | 选错邻近项 | Could：分组或搜索（见 Q4） |

## 10. 待确认

- [仅下拉] **Q1** 是否**完全取消**自由文本？还是「下拉为主 + 高级手填」？  
  - 建议默认：**仅下拉**（与白名单一致）。
- [临时展示「未知字段(原值)」并允许改选] **Q2** 已存 `statField` 不在选项中时：禁用保存并提示修正 / 临时展示「未知字段(原值)」并允许改选 / 其它？
- [列名] **Q3** 结构树摘要行是否改为显示中文，还是继续显示列名？
- [暂时不用] **Q4** 字段约 50+ 项，是否需要分组（按生源/去向/就业地等）或可搜索下拉？（MVP 可先扁平列表）
- [后端字典 API] **Q5** 选项列表由**前端静态字典**提供，还是**后端字典 API**（与白名单同源）？  
  - 产品倾向：只要展示中文、值正确即可；同源维护方式留给技术方案。
- [不用] **Q6** 「单位名称」在导入中有三列同名表头（就业单位 / 职业后单位 / 升学院校）——下拉中文是否需区分（如「单位名称（就业）」「升学院校」），避免选错？

## 11. 验收场景（产品级）

1. 新增指标：下拉选「学院名称」→ 保存成功 → 库/`detail` 中 `statField=college_name`。  
2. 再编辑该指标：下拉显示「学院名称」为选中态。  
3. 未选统计字段点保存：失败提示。  
4. 用含该指标的模版对某校生成报告：该指标按学院分组有数据（在就业数据非空前提下）。  
5. （若 Q2 定为须处理）人为写入非法 `statField` 后打开编辑：按约定策略可见且可改正确字段。
