# 技术方案：指标统计字段下拉选择

**需求来源：** `docs/metric_field_select/PRD分析.md`（一段话需求 + PRD 已确认 Q1–Q6）  
**功能短名：** `metric_field_select`  
**状态：** DRAFT — 仅方案，不编码  
**目标：** 模版指标配置时，「统计字段」以下拉选择（展示中文、值为库列名）；选项与生成侧白名单同源；存储契约 `statField` 不变。

---

## 1. 目标与非目标

### 1.1 目标

| # | 能力 |
|---|------|
| 1 | 后端提供统计字段字典 API（`value`=库列名，`label`=中文） |
| 2 | 字典与 `MetricAggregateService` 可聚合字段**同源**，避免能选不能算 |
| 3 | 管理台 METRIC 表单：文本框改为 `<select>`；仅下拉、不可手填（Q1） |
| 4 | 保存时服务端校验 `statField` ∈ 白名单 |
| 5 | 编辑脏数据：临时展示「未知字段(原值)」并允许改选（Q2） |

### 1.2 非目标

- 不改 `stat_field` 列类型/含义；不新增中文名字段落库
- 不改聚合算法、`valueFormat` / `displayType` / 图表语义
- 不做分组 / 可搜索下拉（Q4）
- 不为三列「单位名称」强制改中文文案（Q6）；靠不同 `value` 区分
- 树摘要继续展示列名，不改中文（Q3）
- 不改 `POST /report/createReport` 外部入参

---

## 2. 现状与切入点

| 能力 | 现状 | 本方案用法 |
|------|------|------------|
| 配置 UI | `admin/index.html` → `metricFormHtml`：`<input id="fStat">` | 改为 `<select id="fStat">`，选项来自字典 API |
| 节点保存 | `ReportTemplateService.applyMetricFields`：非空 + 长度 | **追加**白名单校验 |
| 聚合白名单 | `MetricAggregateService` 内 `FIELD_EXTRACTORS`（私有 Map） | **抽出共享目录**供字典与聚合共用 |
| 中文对照 | `docs/graduate_employment/数据库设计.md` §2 表头 | 作为 `label` 文案权威来源写入目录 |
| 节点 API | `/report-template/node/create|update`、`saveTree` | 入参字段名仍 `statField`；语义收窄为白名单值 |
| 树摘要 | `meta` 拼 `statField` 原文 | **保持列名**（Q3） |

---

## 3. 总体架构

```text
管理台 index.html（METRIC 表单）
        │  GET 字典
        ▼
ReportTemplateController
  GET /report-template/stat-fields
        │
        ▼
GradEmploymentStatFieldCatalog（新建，单一事实源）
  ├── listOptions() → [{ value, label }, ...]   // 字典 API
  ├── isKnown(value) → boolean                  // 保存校验 + 聚合
  └── extractor(value) → Function<Record,String> // 供聚合使用

MetricAggregateService
  └── 改用 Catalog，不再私自维护另一套字段键

ReportTemplateService.applyMetricFields
  └── 非空后：!Catalog.isKnown → IllegalArgumentException
```

**原则：** 配置可选集合 ≡ 聚合可算集合 ≡ Catalog 注册表。增删字段只改一处。

---

## 4. 模块设计

### 4.1 GradEmploymentStatFieldCatalog（新建）

**建议包路径：** `com.myreport.service` 或 `com.myreport.util.employment`（与现有 `service` 风格一致即可，优先 `service`）。

**职责：**

1. 有序注册全部可统计列：`column`（snake_case 库列名）+ `labelZh`（中文）+ `extractor`
2. 对外：
   - `List<Map<String,String>> listOptions()` — 仅 `value`/`label`（或小型 VO）
   - `boolean isKnown(String statField)` — 复用现有 normalize 规则（去 `_`/`-`/空白、小写），与今日 `MetricAggregateService.normalizeFieldKey` 行为一致，保证旧 camelCase 别名仍可识别（若 Catalog 继续注册别名）
   - 供聚合：`Function<GradEmploymentRecord, String> resolveExtractor(String)`

**注册内容：** 与当前 `buildFieldExtractors()` 列集合对齐；`labelZh` 按 `graduate_employment` §2 表头填写。  
三列「单位名称」：`label` 均可为「单位名称」（Q6）；`value` 分别为 `employer_name` / `unit_name_after_occupation` / `further_study_school_name`。

**初始化：** 静态块或私有 `build()` 一次构建不可变 `LinkedHashMap`（保序，便于下拉稳定排序）。排序建议与数据库设计文档章节顺序一致（身份学籍 → 生源 → 去向 → 就业地理 → 单位职业升学）。

**不落库：** 字典为代码内常量，不新建字典表（MVP）。字段变更随发版。

### 4.2 MetricAggregateService（小改）

- 删除（或委托）本地 `FIELD_EXTRACTORS` / `buildFieldExtractors`，改为调用 Catalog
- `isKnownStatField` 委托 `Catalog.isKnown`
- 聚合行为、跳过非法字段策略**不变**

### 4.3 ReportTemplateService（校验收紧）

在 `applyMetricFields` 中，`statField` trim 后：

```text
if blank → 统计字段不能为空
if length > 128 → 既有
if !Catalog.isKnown(statField) → 统计字段不在可选列表中
// 再写 node.setStatField(规范化后的规范列名？见 T1)
```

**规范化落库（推荐）：** 保存时将用户传入值解析为 Catalog 的**规范 snake_case `value`** 再 `setStatField`，避免库内混用 camelCase / 带下划线变体。  
`saveTree` / `node/create` / `node/update` 均走 `applyMetricFields`，一处生效。

### 4.4 ReportTemplateController（新接口）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/report-template/stat-fields` | 返回可选统计字段列表 |

响应风格对齐现有：`code` / `message` / 列表字段（具体字段名留给 API 设计，建议 `fields: [{ value, label }]`）。

无分页、无鉴权增量（与现模版 API 同级）。结果可短缓存（可选，非必须）：字段极少变更，内存 Catalog 已够快。

### 4.5 管理台前端（`static/admin/index.html`）

**加载：** 打开「编辑结构」或首次渲染指标表单前，`GET /report-template/stat-fields` 缓存到 JS 变量（如 `statFieldOptions`）；失败则提示并禁用保存指标。

**`metricFormHtml(n)`：**

```text
<select id="fStat">
  <option value="">请选择统计字段</option>
  <!-- 遍历 options：value=列名，text=中文 -->
  <!-- 若 n.statField 有值且不在 options：追加
       <option value="{原值}" selected>未知字段({原值})</option>  （Q2）
  -->
</select>
```

**保存：** 仍读 `$("fStat").value` 作为 `statField`；空值前端拦截「名称与统计字段必填」。若当前选中为「未知字段」且用户未改选：允许提交时由**后端拒绝**（不在白名单），或前端提示必须改选——**推荐前端保存前若 label 为未知则拦截**，避免无效往返（见 T2）。

**树摘要：** 不改，继续拼列名（Q3）。

**不做：** 手填 input、optgroup 分组、搜索框（Q1/Q4）。

## 5. 数据与契约

### 5.1 持久化

| 项 | 结论 |
|----|------|
| 表结构 | **无变更** |
| `report_template_node.stat_field` | 仍存库列名字符串 |
| 迁移 | 无需；历史合法值可继续回显 |

### 5.2 对外契约变更点

| 变更 | 影响 |
|------|------|
| **新增** `GET /report-template/stat-fields` | 管理台必用；其它调用方可忽略 |
| `node/create`、`node/update`、`saveTree` 中 METRIC 的 `statField` | 从「任意非空串」收窄为「白名单列名」；非法返回 `code=-1` + 可读 message |
| `detail` / 树 JSON | **不变**（仍只返回 `statField` 列名，无 label） |

按仓库规则：点名「做 API 设计」时输出/更新 `docs/` 调用方文档（可增量写在 `docs/metric_field_select/` 或更新 `docs/report_template_config/`）。

## 6. 关键时序

### 6.1 配置人员选字段

```text
打开编辑结构
  → GET /report-template/stat-fields
  → 缓存 options
添加/编辑指标
  → 渲染 select（中文）
  → 用户选择 → value=college_name
  → POST node/create|update（statField=college_name）
  → applyMetricFields + Catalog.isKnown → 落库
```

### 6.2 脏数据编辑（Q2）

```text
detail 中 statField=foo_bar（不在 Catalog）
  → 表单追加 option：未知字段(foo_bar)，selected
  → 用户改选合法项并保存 → 覆盖为合法列名
  → 若坚持保存未知值 → 后端拒绝（或前端先拦，T2）
```

## 8. 与 PRD 对齐摘要

| PRD | 方案落地 |
|-----|----------|
| S-1/S-2 下拉 | `metricFormHtml` → `<select>` |
| S-3 中文/列名 | API `label`/`value`；option text/value |
| S-4 存列名 | 提交与落库仍 `statField` |
| S-5 必填 | 前端空值拦 + 后端 blank 校验 |
| S-6 与可聚合一致 | 单一 Catalog |
| S-7 树摘要中文 | Won't（Q3 列名） |
| C-1 合法回显 | `selected` 匹配 value |
| C-2 未知 | 临时 option「未知字段(原值)」（Q2） |
| Q1 仅下拉 | 无 input |
| Q5 后端字典 | `GET /report-template/stat-fields` |
| Q4/Q6 | 扁平列表；单位名称不强制改 label |

## 9. 风险与缓解

| 风险 | 缓解 |
|------|------|
| Catalog 与文档表头文案不一致 | 编码时按 `graduate_employment` §2 逐列抄 label；单测断言关键列 label |
| 重构 extractor 时漏字段 | 单测：Catalog 选项数 = 原 extractors 数；抽样聚合仍通 |
| 三列同名「单位名称」难辨 | Q6 接受；必要时 option 旁可后续加 `title=列名` 悬停（Could，非 MVP） |
| 收紧校验导致历史脏数据无法「原样保存」 | 符合 Q2：必须改选；列表/生成侧本就会跳过非法字段 |
| 前端字典缓存过期 | 字段随发版变；每次打开编辑结构重新拉取即可 |
