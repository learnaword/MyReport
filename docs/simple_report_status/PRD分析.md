# PRD 分析：简化报告生成状态展示

**需求来源：** 一段话 — 「简化报告生成，增加生成状态，并且不同的状态显示的颜色不一样」

**功能短名：** `simple_report_status`  
**状态：** DRAFT — 仅需求  
**关联能力（现状）：**

- 简化报告配置与运行：`/simple-report/*`（见 `docs/ai_simple_report.md`）
- 运行态字段已存在：`simple_report_run.run_status`（1 待确认 / 2 生成中 / 3 成功 / 4 失败 / 5 取消）
- 管理台：`/admin/index.html#simple-report`
- 对照：管理台「报告管理」列表已有「生成状态」文案，但同样无配色区分

---

## 1. 问题陈述

配置人员在管理台「简化报告」列表中，当前「状态」列展示的是**配置启用/停用**（`simple_report.status`），不是某次出稿的**生成进度/结果**。

用户在 plan → confirm → 异步出 Word 过程中，无法在列表一眼看出：

- 是否还在生成中；
- 最近一次是否成功、失败、待确认或已取消。

若不补「生成状态」及按态配色：

- 只能进弹窗/轮询详情才能感知成败，操作成本高；
- 文案同色挤在一起，失败与成功不易扫读；
- 与「我已经点了确认，报告出了没」这一核心问题错位。

**Primary：**

| 角色 | 场景 |
|------|------|
| 配置 / 运营人员 | 在管理台触发或确认简化报告生成后，需要扫一眼列表知道进度与结果 |
| Agent / API 调用方 | 列表或详情需能读到与 UI 一致的生成态语义（便于轮询展示） |

**JTBD：**

| 场景 | When | I want | so that |
|------|------|--------|---------|

| 辨识 | 多条报告并存、状态不一 | 不同生成态用**不同颜色**区分 | 成功/失败/进行中可一眼扫到 |

- 本期默认不强制改「报告管理」列表配色（可列为可选对齐，见待确认）

## 3. 证据

Assumption — needs validation via 管理台实际操作反馈（一次完整 plan → confirm → 成功/失败）。

对话与现状可观察事实：

- 后端 `SimpleReportRun.runStatus` 五态已落地，详情/运行列表 API 已返回 `runStatus`；
- 配置列表 `toListItem` **未**暴露最近一次运行的生成状态；
- 管理台 `srStatusLabel` 仅映射启用/停用，列表无状态色样式；
- 用户明确要求：增加生成状态 + 不同状态不同颜色。

---

## 4. 假设与成功标准

**Hypothesis：**  
We believe **在简化报告列表（及必要的生成流程界面）展示带配色的生成状态** will **降低「出稿到哪一步了」的认知成本** for **管理台配置人员**.  
We'll know we're right when **确认生成后，无需打开详情即可在列表看到与五态一致的中文标签，且各态颜色可区分**.

| 指标 | 目标 | 如何衡量 |
|------|------|----------|
| 生成态可见 | Must | 列表可见生成状态文案，语义覆盖既有五态（或约定子集，见待确认） |
| 色态一一对应 | Must | 至少成功 / 失败 / 生成中三色可区分；其余态亦有独立或可辨样式 |
| 文案不丢 | Must | 色盲/打印场景下仅看文字仍能理解状态（色是增强，不是唯一信号） |
| 与库态一致 | Must | UI 展示与 `run_status`（或约定的「列表聚合态」）一致，无自造第三套枚举 |
| 启用/停用仍可辨 | Should | 配置启用态与生成态不互相覆盖、不混淆（分列或主次清晰） |

## 5. 方案概述（What）

```text
简化报告（配置）列表 / 生成相关界面
├── 生成状态（来自最近一次相关运行，或当前进行中的运行 —— 聚合规则见待确认）
│   ├── 待确认
│   ├── 生成中
│   ├── 成功
│   ├── 失败
│   └── 取消
└── 展示：中文标签 + 按态固定颜色（徽章/色字均可，产品上要求「不同状态颜色不一样」）

可选并列保留：
└── 配置状态：启用 / 停用（现有 status）
```

**边界：**

| 做 | 不做（本期） |
|----|----------------|
| 管理台简化报告侧展示生成状态 + 配色 | 新建与 `run_status` 无关的另一套状态机 |
| 列表/弹窗中状态可读且与后端一致 | 强制重做邮件/落盘/出稿内核 |
| 失败态可辅助看到失败信息入口（如 title/提示） | 完整实时大屏、WebSocket 推送（轮询是否增强见待确认） |

**产品倾向（基于现有能力，非实现拍板）：**  
生成状态语义复用已有 `run_status`；本期重点是**暴露与展示**，不是再发明一套状态。

## 6. 功能需求

### 6.1 生成状态语义（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| S-1 | 生成状态取值与现有运行态对齐：待确认 / 生成中 / 成功 / 失败 / 取消 | Must | 数值约定已存在：1/2/3/4/5 |
| S-2 | 列表中每条简化报告配置须能展示一个「当前/最近生成状态」 | Must | 「取哪一次 run」见待确认 Q1 |
| S-3 | 无任何运行记录时，有明确空态文案（如「未生成」） | Must | 空态也需可辨颜色或中性色，见待确认 |
| S-4 | 失败时可感知失败原因入口（悬停/详情已有 failMessage 即可） | Should | |

### 6.2 管理台展示与配色（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| U-1 | 简化报告列表增加或改造「生成状态」展示 | Must | 与现「状态=启用/停用」区分，见 Q2 |
| U-2 | 不同生成状态使用不同颜色展示 | Must | 建议色板见 §7；最终色值可调，但须态与色一一对应 |
| U-3 | 状态须同时有中文文案，禁止仅靠颜色表达 | Must | 可访问性 |
| U-4 | plan/confirm 弹窗或生成流程中，当前 run 的状态也应配色一致 | Should | 与列表同源映射函数 |
| U-5 | 生成中时，用户无需手刷整页也能在合理时间内看到终态 | Should | 现有轮询可复用；是否列表自动刷新见 Q4 |

### 6.3 调用方可读（Should）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| A-1 | 配置列表接口返回生成状态相关字段（或已有字段补齐），供管理台/外部一致消费 | Should | 具体字段名留给 API 设计；PRD 只要求「列表能拿到生成态」 |
| A-2 | 字段语义与 `runs/detail` 的 `runStatus` 一致，避免两套中文映射冲突 | Must | |

### 6.4 明确不做（Won't）

| 项 | 原因 |
|----|------|
| 为配色单独新建状态枚举表/字典服务 | 五态已够用 |
| 本期强制改造报告管理列表配色 | 非本需求点名；可作可选对齐 |
| 用进度百分比替代离散生成状态作为主展示 | 用户要的是状态 + 颜色；进度仍可在 Redis 侧存在 |

## 7. 状态与颜色（产品建议，待确认后定稿）

| 生成状态 | 建议文案 | 建议色意 | 说明 |
|----------|----------|----------|------|
| （无运行） | 未生成 | 中性灰 | 从未 plan/confirm |
| 待确认 | 待确认 | 警告黄/橙 | 已有 plan，等人确认 |
| 生成中 | 生成中 | 信息蓝 | 异步出稿中 |
| 成功 | 成功 | 成功绿 | 可下载 |
| 失败 | 失败 | 危险红 | 可看失败原因 |
| 取消 | 已取消 | 中性灰（可略深于未生成） | 待确认被取消 |

> 具体色值（hex）属视觉实现细节，技术方案/编码时定；PRD 只锁定「态 ↔ 色意」一一对应且可区分。

## 8. 用户流程（Happy Path）

1. 用户打开管理台「简化报告」列表。  
2. 看到每条报告的**生成状态**（带颜色）：如「未生成」或上次「成功」。  
3. 用户对某条执行生成计划并确认 → 状态变为「生成中」（蓝）。  
4. 生成结束 → 「成功」（绿）可下载，或「失败」（红）可查看原因。  
5. 若停在计划未确认 → 「待确认」（黄/橙），与终态可区分。

## 10. 风险

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| 「配置启用状态」与「生成状态」同列混淆 | High | Med | 分列命名；PRD 要求不互相覆盖 |
| 「最近一次」聚合规则不清（成功稿 vs 进行中） | Med | High | Q1 拍板：优先展示进行中/待确认，否则最近一次 |
| 仅改 UI 不改列表 API 导致无法展示最新态 | Med | High | 列表须能拿到生成态字段（A-1） |
| 颜色对比度不足或仅靠色 | Low | Med | 文案必带；色意遵循绿/红/蓝/黄习惯 |
