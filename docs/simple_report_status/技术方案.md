# 技术方案：简化报告生成状态展示

**需求来源：** `docs/simple_report_status/PRD分析.md`  
**功能短名：** `simple_report_status`  
**状态：** DRAFT — 仅方案，不编码  
**目标：** 在简化报告**配置列表**与生成相关 UI 中展示与 `run_status` 一致的**生成状态**，并按态配色；列表 API 补齐可读字段。不改状态机、不改出稿内核。

---

## 0. PRD 决策吸收（标注项）

| 题 | 方案采用 |
|----|----------|
| Q1 列表取哪条 run | **最近一次任意态**（`create_time DESC`）：避免历史「待确认」盖住更新的成功/失败；生成中自然是最新一条时仍会显示「生成中」 |
| Q2 启用列 | **A**：保留「配置状态」（启用/停用），**另增「生成状态」列** |
| Q3 色意 / 对齐报告管理 | 采用 PRD §7 色意；**本期不对齐**报告管理列表配色（可后续复用同一 CSS 类） |
| Q4 列表轮询 | **轻量**：弹窗内保持现有 `runs/detail` 轮询；列表在**当前页存在「生成中」**时每隔约 3s 自动 `loadSimpleReports` 一次，否则仅手动「刷新」；confirm 成功后立即刷新列表 |
| Q5 未生成 | **是**：`latestRunStatus = null` 展示「未生成」+ 中性灰 |
| Q6 调用方文档 | **是**：列表契约增量 → 更新 `docs/ai_simple_report.md`（或增量短文 `docs/simple_report_status.md` + 链到主文档） |

| 能力 | 现状 | 本方案用法 |
|------|------|------------|
| 运行态 | `SimpleReportRun.runStatus`：1/2/3/4/5 | **唯一事实源**；UI/API 只映射展示 |

## 3. 总体设计

```text
GET /simple-report/list
  → SimpleReportService.toListItem(r)
       → resolveDisplayRun(reportId)   // 聚合规则 §4
       → 写入 latestRunStatus / latestRunId / latestFailMessage …

管理台 loadSimpleReports
  → 「配置状态」列：启用/停用（原 status）
  → 「生成状态」列：badge(latestRunStatus)  色+文
  → 若页内存在 status=2 → setInterval 刷新列表

弹窗
  → 复用同一 badge 函数渲染 detail.runStatus
```

**原则：**

1. **不发明第三套枚举**；API 数字与库一致，文案仅在前端（或两端同表，见 §5）。  
2. **下载字段与展示字段分离**：`canDownload` / `latestSuccessRunId` 仍只看 SUCCESS 可下载稿；`latestRunStatus` 按 §4 展示规则，允许为待确认/失败等。  
3. **色是增强，文案必带**。

## 4. 「列表展示用」run 聚合规则

对单个 `reportId`：

```text
1. 查该配置下任意 run，按 create_time DESC, id DESC 取 1 条
   → 若有，即为展示 run
2. 否则无 run → latestRunStatus = null（未生成）
```

**理由：** 展示「最近一次操作结果」。若优先任意「待确认」，历史未确认计划会盖住之后已成功的出稿，造成「文件已生成仍显示待确认」。

## 5. API 增量（契约摘要）

> 详细字段表留给「做 API 设计」；此处锁定方案层约定。

### 5.1 `GET /simple-report/list` 每条 `content[]` 增量

| 字段 | 类型 | 说明 |
|------|------|------|
| `latestRunStatus` | Integer \| null | 展示用 `run_status`；无 run 为 `null` |
| `latestRunId` | Long \| null | 对应 run id |
| `latestFailMessage` | String \| null | 仅当展示态为失败时建议带回，便于 `title` 悬停；其它态可省略或 null |

**保留不变：** `status`（配置启用）、`canDownload`、`latestSuccessRunId`。

**可选（Should）：** `detail` 同步带上上述三字段，避免编辑弹窗再猜。

**不改：** `runs/detail`、`runs/list` 的 `runStatus` 语义。

### 5.2 状态文案（前后端宜共用同一张表）

| latestRunStatus | 文案 |
|-----------------|------|
| `null` | 未生成 |
| 1 | 待确认 |
| 2 | 生成中 |
| 3 | 成功 |
| 4 | 失败 |
| 5 | 已取消 |

## 6. 后端实现要点

### 6.1 涉及类（预计）

| 类 | 改动 |
|----|------|
| `SimpleReportRunRepository` | 增加查询：① `findByReportIdAndRunStatusInOrderByCreateTimeDescIdDesc`；② `findByReportIdOrderByCreateTimeDescIdDesc`（Pageable top1）；或批量 `findByReportIdInOrderByCreateTimeDescIdDesc` |
| `SimpleReportService` | `resolveDisplayRun`；`toListItem`（及可选 detail）填新字段；列表页避免 N+1（见 §6.2） |
| Controller | 无路径变更；响应 JSON 多字段即可 |

**无**新 Entity / 无 DDL。

### 6.2 列表性能

当前 `toListItem` 已对每行调用 `findLatestDownloadableRun`（最多约 20 次/页）。本方案再查展示 run，若 naive 每行 1～2 次查询，页大小 20 可接受但偏糙。

**推荐（本期）：**

1. 取出本页全部 `reportId`；  
2. **一次** `findByReportIdInOrderByCreateTimeDescIdDesc(ids)`（或分批）；  
3. 内存按 `reportId` 分组，各组内按 §4 规则选出 display run；  
4. `canDownload` / `latestSuccessRunId` 可继续走现有方法，或同批数据中筛 SUCCESS+可下载（二次优化，非必须）。

若批量查询实现成本高，**可接受兜底**：每行 `PageRequest.of(0,1)` 各查一次活跃态 + 一次最近态（与现网下载查询同量级）。

### 6.3 失败信息

- 展示态为 4 时：`latestFailMessage` = 该 run 的 `failMessage`（截断与库一致，≤512）。  
- 其它态：`null`。  
- UI：徽章 `title` 属性展示失败原因。

## 7. 管理台 UI

### 7.1 表头

| 原列 | 调整 |
|------|------|
| 状态 | 改名为 **配置状态**（启用/停用） |
| （新增） | **生成状态**（色徽章 + 文案） |

列顺序建议：`ID | 名称 | 交付目录 | 区块数 | 配置状态 | 生成状态 | 更新时间 | 操作`。

### 7.2 配色（CSS 类，色值实现时可微调）

| 态 | class 建议 | 色意 |
|----|------------|------|
| 未生成 | `sr-gen-none` | 中性灰 |
| 待确认 | `sr-gen-pending` | 黄/橙 |
| 生成中 | `sr-gen-running` | 信息蓝 |
| 成功 | `sr-gen-ok` | 成功绿 |
| 失败 | `sr-gen-fail` | 危险红 |
| 已取消 | `sr-gen-cancel` | 深灰 |

形态：圆角小徽章（padding + 浅底 + 深字），**禁止**只用改字体颜色而无文案。

### 7.3 JS

```text
srConfigStatusLabel(status)     // 原 srStatusLabel：启用/停用
srGenerateStatusBadge(runStatus) // 返回带 class 的 HTML 或 DOM
```

弹窗内展示当前 `runStatus` 时调用同一 `srGenerateStatusBadge`。

### 7.4 刷新策略

| 时机 | 行为 |
|------|------|
| 进入 panel / 点刷新 / 翻页 | `loadSimpleReports` |
| confirm 返回成功 | 立即 `loadSimpleReports`（列表可见「生成中」） |
| 当前页 `content` 中存在 `latestRunStatus === 2` | `setInterval` ~3s 刷新；离开 panel 或页内无生成中则 `clearInterval` |
| 弹窗轮询到成功/失败 | 现有逻辑 + 顺带刷新列表 |

## 8. 数据流（Happy Path）

```text
用户 confirm
  → run_status=2
  → 列表刷新 → 蓝色「生成中」
  → Spire 成功回调 → run_status=3
  → 列表轮询/刷新 → 绿色「成功」+ 可下载

用户仅 plan 未 confirm
  → run_status=1 → 列表黄色「待确认」

从未 plan
  → latestRunStatus=null → 灰色「未生成」
```

## 10. 风险与缓解

| Risk | 缓解 |
|------|------|
| 启用态与生成态混淆 | 列名「配置状态 / 生成状态」；不同映射函数 |
| N+1 查询 | §6.2 批量优先 |
| 列表展示成功但文件已删 | `canDownload` 仍做文件校验；状态可为成功而按钮不出现（与现网一致） |
| 轮询过频 | 仅「页内有生成中」才轮询；间隔 ≥3s；切 panel 清理 |
