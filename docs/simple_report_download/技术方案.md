# 技术方案：简化报告下载

**需求来源：** 一段话「简化报告支持下载」  
**功能短名：** `simple_report_download`  
**状态：** DRAFT — 仅方案，不编码  
**关联：** `docs/ai_simple_report*`、`docs/report_download.md`（管理台报告下载范式）

---

## 0. 结论摘要

后端 **`GET /simple-report/runs/download?runId=` 已存在**，可按运行实例下载 `.docx`。缺口主要在：

1. **管理台无下载入口**（列表仅「编辑 / 删除」；生成成功后无可点下载）
2. **配置列表无 `canDownload` / 最近成功 run 提示**，用户不知道能否下
3. **下载解析偏弱**：未做物理根路径穿越校验；附件名用磁盘文件名而非报告名；失败态与 managed 不完全对齐

本方案：**复用现有按 `runId` 下载契约**，补齐「配置维度最近成功稿」能力与管理台按钮，并与 `managed-report/download` 安全/体验对齐。不新建下载中心，不改 Spire 落盘主路径。

---

## 1. 目标与非目标

### 1.1 目标

| # | 能力 |
|---|------|
| 1 | 管理台简化报告：**可下载已成功生成的 Word** |
| 2 | 列表/详情暴露 **`canDownload`**（及可选 `latestSuccessRunId`），按钮可灰可显 |
| 3 | 支持两种入口：**按配置下最近成功稿**、**按 runId 下指定稿**（后者已有，加固即可） |
| 4 | 下载安全：禁止客户端传路径；canonical 落在 `prefixFilePhysicalPath` 下 |
| 5 | 附件文件名优先用报告名称（sanitize + `.docx`） |

### 1.2 非目标

- 不新做独立「下载中心」产品页
- 不改 plan → confirm → 异步生成主流程
- 不把交付目录做成对外静态直链（仍走鉴权同级的 admin / API 附件流）
- 本期不做历史 run 列表独立 UI（可在编辑弹窗成功态提供「下载本次」即可）
- 不引入新表、不迁移历史数据

---

## 2. 现状与缺口

| 能力 | 现状 | 缺口 |
|------|------|------|
| 按 run 下载 | `SimpleReportController.download` → `resolveDownloadFile(runId)` | 仅 `STATUS_SUCCESS`；无路径穿越校验；无友好 downloadName |
| Run 列表/详情 | `toRunMap` 含 `filePath`/`deliveryPath`/`runStatus` | **无 `canDownload`** |
| 配置列表 | `toListItem` 仅配置元信息 | **无能否下载 / 最近成功 runId** |
| 管理台 | `#simple-report`：编辑弹窗可 plan/confirm/轮询 | 成功仅文案提示路径；**无「下载」按钮**；列表无下载 |
| 对照范式 | `managed-report`：`canDownload` + `GET /download?id=` + 前端 blob 保存 | 简化报告应对齐交互，参数语义不同（配置 id / runId） |

**可下载路径优先级（保持现网）：** `deliveryPath` 非空优先，否则 `filePath`。

---

## 3. 总体方案

```text
管理台 #simple-report
  ├─ 配置列表「下载」──► GET /simple-report/download?id={配置id}
  │                         （解析该配置最近一次 SUCCESS run → 附件流）
  └─ 编辑弹窗「下载本次」─► GET /simple-report/runs/download?runId=
                            （已有；成功轮询后显示）

SimpleReportService
  ├─ canDownload(run) / findLatestSuccessRun(reportId)
  ├─ prepareDownloadByRunId / prepareDownloadByReportId
  └─ 路径校验 + sanitize 文件名（对齐 ManagedReportService）
```

**推荐默认行为：** 配置列表「下载」= 该配置 **最近一次** `runStatus=SUCCESS` 且文件仍在的稿（按 `finishedTime`/`id` 降序）。编辑弹窗在本次 run 成功后提供「下载本次」，避免用户搞不清下的是哪次。

## 4. 后端设计

### 4.1 契约增量（相对现网）

| 方法 | 路径 | 说明 | 变更 |
|------|------|------|------|
| GET | `/simple-report/runs/download?runId=` | 按运行下载 | **加固**（路径校验、文件名、失败语义） |
| GET | `/simple-report/download?id=` | 按配置下载最近成功稿 | **新增** |
| GET | `/simple-report/list` | 配置分页 | 增加 `canDownload`、`latestSuccessRunId`（可选） |
| GET | `/simple-report/runs/list`、`/runs/detail` | 运行查询 | 增加 `canDownload` |

> 编码落地时按仓库规则同步更新 `docs/ai_simple_report.md` 与/或本功能调用方文档（如 `docs/simple_report_download.md`）。

### 4.2 `canDownload` 规则

对 **单次 run**：

```text
canDownload = (runStatus == SUCCESS)
            && 有效路径存在（deliveryPath 优先，否则 filePath）
            && File.isFile()
            && 路径在物理根目录下
```

对 **配置列表**：

```text
canDownload = 存在至少一条满足上述条件的 SUCCESS run
latestSuccessRunId = 最近一条的 runId（无可下则为 null）
```

**与 managed 的差异（方案默认）：**

| 项 | managed-report | 本方案（简化报告） |
|----|----------------|-------------------|
| 失败仍有旧稿 | 允许下旧稿 | **默认不允许**（仅 SUCCESS）；简化报告失败时常无可靠旧稿绑定在配置上 |
| 主键 | 报告实例 id | 配置 id **或** runId 两套入口 |

### 4.3 下载解析（伪代码）

```text
prepareByRunId(runId):
  run = requireRun(runId)
  assert canDownload(run) else throw "报告尚未生成成功" / "文件不存在…"
  file = resolvePath(run)   // deliveryPath ?: filePath
  assertUnderPhysicalRoot(file)
  name = sanitize(run.reportName) + ".docx"
  return (file, name)

prepareByReportId(reportId):
  report = requireReport(reportId)  // 未删
  run = findLatestSuccessRunnable(reportId)  // 有文件的最近 SUCCESS
  if run == null: throw "暂无可下载文件"
  return prepareByRunId(run.id)
```

Repository 建议（实现时二选一即可）：

- `findFirstByReportIdAndRunStatusOrderByFinishedTimeDescIdDesc(reportId, SUCCESS)`，再在 Service 侧过滤文件存在；或
- 查最近若干条 SUCCESS，取第一条 `canDownload==true`。

### 4.4 Controller 响应

与现网 / managed 一致：

- **成功：** HTTP 200，`Content-Type` 为 docx，`Content-Disposition: attachment; filename=...; filename*=UTF-8''...`
- **业务失败：** 现网 simple-report 多用 **HTTP 200 + JSON `{code:-1,message}`**；managed 部分用 400。  
  **方案默认：保持 simple-report 现网「200 + JSON」**，避免打断已有 Skill/curl；管理台前端已按 Content-Type 区分（见 managed 的 blob 逻辑，可复用）。

### 4.5 安全

- 禁止请求体/参数传磁盘路径
- `getCanonicalPath()` 必须落在 `FileUtil.fileConfig.getPrefixFilePhysicalPath()` 下
- 软删配置：`/download?id=` 拒绝；已有 run 的 `/runs/download` **方案默认仍允许**（历史稿可追溯；若需连带禁止标「待确认」）
- 文件名 sanitize：去路径分隔符、控制字符；空名回退 `simple-report-{runId}.docx`

### 4.6 库表

**无 DDL。** 继续用 `simple_report_run.delivery_path` / `file_path` / `run_status`。

## 5. 管理台设计

### 5.1 配置列表

操作列在「编辑」旁增加 **「下载」**：

- `row.canDownload === true` 时渲染按钮，否则不渲染（或 disabled + title 说明）
- 点击：`GET /simple-report/download?id=` + 复用 `downloadManagedReport` 同款 blob 保存逻辑（可抽 `downloadDocx(url, msgEl)`）

### 5.2 编辑弹窗

在「确认生成 Word」轮询到 `runStatus===3` 后：

- 提示成功文案保留
- 显示 **「下载本次」** → `/simple-report/runs/download?runId={srPendingRunId}`
- 若打开编辑时回显的最近 run 已是 SUCCESS，也可显示「下载最近成功稿」（可选，编码时可做最小：仅本次成功后出现）

### 5.3 不做

- 不恢复已去掉的列表「计划」按钮
- 本期不单独做「运行历史」全页

## 8. 测试要点

| 场景 | 期望 |
|------|------|
| SUCCESS + delivery 文件在 | `/runs/download` 与 `/download?id=` 均可下 |
| 仅 filePath、无 delivery | 仍可下 |
| PENDING / GENERATING / FAILED | `canDownload=false`；下载返回业务失败 JSON |
| 配置从未成功 | 列表无下载按钮；`/download?id=` 失败 |
| 多次 SUCCESS | `/download?id=` 取最近一次 |
| 路径在根外 / 文件已删 | 失败，不泄露绝对路径细节给 message |
| 中文报告名 | Content-Disposition 可正确触发浏览器文件名 |
| 管理台 | 列表与弹窗均可保存 `.docx` |

编译：`mvn -DskipTests compile`。

## 9. 风险与回滚

| 风险 | 缓解 |
|------|------|
| 新增 `/download` 与未来其它模块路径冲突 | 挂在已有 `/simple-report` 下，与 managed 对称 |
| 列表每次算 canDownload 多查库 | 按页批量查最近 SUCCESS（或一次 IN reportIds 取 max）；页 size 小（20）可接受 |
| Skill 依赖「仅 SUCCESS」 | 与现网一致，无破坏 |
| 加固路径后历史异常路径不可下 | 符合安全预期；提示重新生成 |

回滚：去掉新接口与前端按钮即可；`runs/download` 行为可保留加固或回退。
