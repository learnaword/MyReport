# PRD 分析：AI 简化报告

**需求来源：** 一段话 + 多轮澄清 —  
1）用 Agent / Skill（类 OpenClaw「龙虾」）按用户诉求或定时任务，先梳理一份 MD（查什么数据、接口是什么、渲染成什么图）；  
2）配置侧：配置数据接口 + 渲染样式（如柱状图），交给 AI，最终要一份 Word；  
3）一份 Word 可配置**多个**数据接口（多区块）；  
4）结果可发送邮件 / 入库 / 存某目录；  
5）出 Word 不依赖 Word MCP，优先复用现有报告生成能力。

**功能短名：** `ai_simple_report`  
**状态：** DRAFT — 仅需求  
**关联能力（现状）：**

- Word 异步生成：`POST /report/createReport`（消费 `reportJsonArr`）
- 模版树 + 指标聚合 + 管理台一键生成：`report_template_config` / `report_manage`
- Agent 侧：OpenClaw / Comate 类 Skill 编排（本仓库外或旁路 Skill 包）

---

## 1. 问题陈述

现有报告能力偏「模版树 + 就业指标聚合」的完整链路，配置与理解成本高。业务侧常有更轻量诉求：

- 指定若干**外部/内部数据接口**；
- 为每个接口指定**渲染样式**（柱状图、表格等）；
- 希望 **Agent** 根据诉求（或定时任务）自动组织内容并产出 **一份 Word**。

若不做「简化报告」：要么继续手工拼 `reportJsonArr`，要么让龙虾用 `python-docx` 另起一套出稿，与现有图表/模版/落盘/进度体系割裂，难复用、难统一运维。

## 4. 假设与成功标准

**Hypothesis：**  
We believe **「多接口 + 样式配置 + Agent 编排 MD + 复用现有 Word 生成」** will **降低轻量报告出稿成本** for **配置人员与 Agent 使用者**.  
We'll know we're right when **配置 N 个接口后，一次触发能稳定产出含多区块的一份 `.docx`，且交付方式可按配置到达**.

| 指标 | 目标 | 如何衡量 |
|------|------|----------|
| 多接口一份 Word | Must | 同一报告配置 ≥2 个接口，生成文档内对应 ≥2 个数据区块 |
| 样式可配 | Must | 至少支持柱状图；每个接口可独立指定渲染样式 |
| Agent 可编排 | Must | 触发后产出结构化 MD（查询项列表 + 接口 + 样式 + 交付） |
| 可出 Word | Must | 最终得到可下载/可落盘的 `.docx`（异步进度可感知，与现有约定一致为佳） |
| 交付可达 | Should | 至少一种：邮件 / 入库 / 目录；其余可分期 |
| 计划可审 | Should | MD 可人工查看后再执行（或定时跳过审阅，见待确认） |

## 5. 方案概述（What）

```text
简化报告配置
├── 报告元信息（名称、触发方式：对话诉求 / 定时 …）
└── 区块列表（1..N）
    ├── 数据接口（URL / 方法 / 参数约定 …）
    ├── 渲染样式（柱状图 / 表格 / …）
    └── （可选）标题、说明文案提示

触发（用户诉求 | 定时任务）
    ↓
Agent / Skill
    ├── 读取配置 + 诉求
    ├── 产出结构化 MD（1 查什么 → 接口；2 …；交付方式）
    └── 执行：按区块拉数 → 按样式组装 → 触发出 Word
    ↓
一份 .docx
    ↓
交付：邮件 / 入库 / 指定目录（按配置）
```

**边界：**

| 做 | 不做（本期） |
|----|----------------|
| 多接口多区块配置语义 | 重做 Spire 渲染内核 |
| Agent 编排 MD + 触发出稿 | 强制依赖 Word MCP |
| 复用或薄封装现有 Word 生成 | 用龙虾本地 python-docx 作为主出稿路径（可作为 fallback，见待确认） |
| 交付通道（至少一种落地） | 完整 BI 交互大屏 |

**与「龙虾怎么出 Word」的产品选择（已倾向）：**  
龙虾侧 Skill 负责**编排与调用**；**真正写 Word** 走 MyReport（或等价服务），避免双引擎。

## 6. 功能需求

### 6.1 配置：一份 Word ↔ 多接口（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| C-1 | 一份简化报告可绑定 **多个**数据接口配置项（有序） | Must | 顺序即文档区块顺序（默认） |
| C-2 | 每个接口项可独立配置 **渲染样式** | Must | 首期至少「柱状图」；表格为强烈建议 |
| C-3 | 接口配置需能表达「如何取数」的最小信息 | Must | 如地址、方法、必要参数；鉴权方式见待确认 |
| C-4 | 可增删改区块、调整顺序 | Should | |
| C-5 | 配置可持久化并再次用于 Agent / 定时 | Must | |

### 6.2 Agent 编排与 MD（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| A-1 | 支持按**用户诉求**触发编排 | Must | 自然语言 + 已有配置 |
| A-2 | 支持按**定时任务**触发编排/出稿 | Should | 调度载体见待确认 |
| A-3 | 编排产出结构化 **MD**，至少包含：查询项序号、查什么、接口是什么、渲染成什么、交付方式 | Must | 可给人审阅 |
| A-4 | MD 与配置区块一一对应或可映射 | Must | 避免漏块、错块 |
| A-5 | Skill 形态可在 OpenClaw / 同类 Agent 中安装使用 | Should | Skill 包可放仓库旁路目录，见后续技术方案 |

### 6.3 拉数、渲染、合成 Word（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| G-1 | 按 MD/配置顺序调用各接口取数 | Must | 失败策略见待确认 |
| G-2 | 按各区块样式渲染（柱状图等）并写入**同一** Word | Must | |
| G-3 | 出稿异步可感知进度/成败 | Should | 对齐现有 Redis 进度更佳 |
| G-4 | 不依赖 Word MCP | Must | 产品已确认倾向 |

### 6.4 交付（Should / Must 分期）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| D-1 | 生成后保存到**指定目录** | Must | 路径规则见待确认 |
| D-2 | 生成后**入库**（报告实例/文件元数据） | Should | 可复用报告管理实体或新表 |
| D-3 | 生成后**发送邮件** | Could | 收件人、附件大小限制见待确认 |

### 6.5 明确不做（Won't）

| 项 | 原因 |
|----|------|
| Word MCP 作为主出稿路径 | 已有 Spire 链路；MCP 非必需 |
| 本期做成完整模版树编辑器的替代品 | 简化报告与完整模版并存 |
| 保证「设计图 / UI 图 / 交互图」均达到专业设计工具水准 | 名称可映射为：示意图/静态图/表；能力边界见待确认 |
| Agent 无配置纯幻觉编造数据当正式稿 | 数据须来自已配置接口（除非明确「演示模式」） |

## 8. 用户流程（Happy Path）

1. 配置人员创建「简化报告」，添加接口 A（柱状图）、接口 B（表格）… 并保存。  
2. 用户对 Agent 说：「按 XX 简化报告配置出一份本周概况」；或定时任务到期。  
3. Agent 读取配置 + 诉求 → 写出 MD 计划（可展示）。  
4. 执行：依次调接口 → 按样式组装 → 触发生成 → 得到一份 Word。  
5. 按交付配置：写入目录 / 记库 /（可选）发邮件。

## 10. 风险

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| 外部接口形态不一，Agent/组装难通用 | High | High | MVP 限定接口响应契约（JSON 白名单形状）；超出标「待适配」 |
| 双路径出 Word（龙虾 python-docx vs MyReport）导致版式分裂 | Med | High | PRD 已倾向单路径：MyReport 出稿 |
| 「交互图」期望与 Word 静态载体冲突 | Med | Med | 产品上定义为静态嵌入；交互留给管理台预览（若需要） |
| 定时 + 邮件涉及运维与密钥 | Med | Med | 分期；密钥走环境变量 |
| 与现有 `report_manage` 概念重叠，用户困惑 | Med | Low | 命名区分「简化报告」vs「模版报告」；入口分开 |

| 龙虾常见做法 | 本需求选择 |
|--------------|------------|
