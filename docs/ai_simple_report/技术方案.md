# 技术方案：AI 简化报告

**需求来源：** `docs/ai_simple_report/PRD分析.md`  
**功能短名：** `ai_simple_report`  
**状态：** DRAFT — 仅方案，不编码  
**目标：** 配置「多数据接口 + 渲染样式」→ Agent/人工触发先出可审 MD 计划 → 确认后拉数组装 → **复用现有 Spire 链路**合成一份 `.docx` → MVP 落指定目录；本服务提供可被 Agent 调用的编排 API；定时用 Quartz。

## 1. 目标与非目标

### 1.1 目标

| # | 能力 |
|---|------|
| 1 | 简化报告配置 CRUD：名称、交付目录、有序区块（接口 + 样式 + 标题） |
| 2 | 编排：根据配置 + 用户诉求生成结构化 **MD 计划**，状态待确认 |
| 3 | 确认执行：按区块 HTTP 拉数 → 归一化 → 组装 `reportJsonArr` → `SpireReportUtil.createReport` |
| 4 | 成功后将 `.docx` **复制/落到配置目录**；进度仍走 Redis |
| 5 | Quartz 按 cron 触发「生成计划」；是否自动确认见 §8.3 |
| 6 | 旁路 OpenClaw `SKILL.md`：教 Agent 调本服务 API（不出本地 python-docx） |

### 1.2 非目标

- 不引入 Word MCP；不以龙虾 `python-docx` 为主出稿路径
- 不重写 Spire/POI；不替代 `report_manage` 就业模版主路径
- MVP 不做接口鉴权、邮件、独立下载中心产品化
- 不做交互式网页图表
- 不让 Agent 幻觉数据充当正式稿（数据必须来自配置接口）

## 2. 现状与切入点

| 能力 | 现状 | 本方案用法 |
|------|------|------------|
| Word 生成 | `SpireReportUtil.createReport`；落盘约 `files/.../report/{name}_{id}.docx`；Redis `report:list:%d` / `report:lock:%d` | 组装完后**内聚调用**；`reportId` 用「运行实例 id」并解决与 managed 冲突（§6.4） |
| 静态指标节点 | `nTreeType=1`，`nType=0`，`strData` + `displayType`/`chartStyle`；`TemplateUtil` 将 BAR/PIE/LINE → `lChartexample` | 每区块 → 一个静态 METRIC 节点 |
| 组装参考 | `ReportJsonAssembler` + `ManagedReportService.generate` | **新建** `SimpleReportAssembleService`，扁平「标题包指标」树，不复用就业聚合 |
| HTTP 出站 | 仅有下载图等 `HttpURLConnection` | 新建 `SimpleReportHttpFetcher`（超时、大小上限、JSON 解析） |
| 管理台 | `/admin/index.html` hash panel | 新增 `simple-report` panel |
| 定时 | 无业务 Quartz | 引入 Spring Quartz 或 `@Scheduled`+cron 配置（优先轻量，见 §8） |

**引擎节点要点（复用，不改语义）：**

| 角色 | 字段 |
|------|------|
| 标题 | `nTreeType=0`，`strTitle`，`strContent`，`children` |
| 指标（静态） | `nTreeType=1`，`nType=0`，`strTitle`，`strData`（JSON 字符串），`displayType`=`TABLE`\|`CHART`，`chartStyle`=`BAR`\|`PIE`\|`LINE` |

`strData` 内与现网一致，至少：

```json
{
  "strText": "",
  "strText2": "",
  "headList": [["名称", "数值"]],
  "tableList": [["分组A", "12"], ...],
  "chartDataList": [["名称", "数值"], ["分组A", "12"], ...],
  "title": "区块标题",
  "cellHeadList": [],
  "cellTableList": []
}
```

## 3. 总体架构

```text
管理台 simple-report panel          OpenClaw Skill（薄）
        │                                    │
        ▼                                    ▼
SimpleReportController  (/simple-report/*)
        │
        ├── 配置 CRUD（SimpleReport + blocks）
        ├── POST .../plan      → 写 MD，状态 PENDING_CONFIRM
        ├── POST .../confirm   → 异步 execute
        ├── GET  .../runs/{id} → 状态 / 计划 MD / 文件路径
        └── （Quartz Job 调同一套 plan/confirm 策略）

execute 流水线（SimpleReportRunService）
  1. 读配置 + 已确认 plan
  2. 按 blocks 顺序 HttpFetcher 拉数 → 归一化为 head/table/chart
  3. SimpleReportAssembleService → reportJsonArr + overallSetting
  4. SpireReportUtil.createReport(reportId=runId映射)
  5. 成功回调：复制到 deliveryDir；更新 run 状态
```

**原则：**

- Agent **不写 Word**；只调 API、展示 MD、代用户点确认（或提示人确认）。
- 拉数与组装在 **MyReport 进程内**完成，保证契约与图表样式可控。
- 与 `managed_report` **实体分离、入口分离**；共享的只有 Spire + Redis 进度机制。

## 4. 模块设计

### 4.1 持久化（逻辑模型；DDL 细节留给「数据库设计」）

| 实体 | 职责 |
|------|------|
| `SimpleReport` | 名称、启用状态、`delivery_dir`、软删、创建时间 |
| `SimpleReportBlock` | 归属报告、`sort_order`、标题、`http_method`、`url`、`query_json`/`body_json`（可空）、`render_style`、`prompt_hint` |
| `SimpleReportRun` | 一次运行：关联配置快照或 configId、用户诉求原文、`plan_md`、状态机、错误摘要、`engine_report_id`、`file_path`、`delivery_path` |

**状态机（Run）：**

```text
PLANNING → PENDING_CONFIRM → GENERATING → SUCCESS
                 │                │
                 │                └→ FAILED
                 └→ CANCELLED（可选）
```

- `plan`：写入 `plan_md`，进入 `PENDING_CONFIRM`
- `confirm`：仅 `PENDING_CONFIRM` 可进入 `GENERATING`
- Spire 成功/失败回调更新终态（对齐 `ManagedReportGenerateSync` 模式）

**配置快照：** 执行时把当时 blocks 序列化进 Run（JSON），避免确认后改配置导致结果漂移。

### 4.2 SimpleReportService（配置）

- CRUD 报告；blocks 整树替换或按序保存
- 校验：`url` 非空、方法 GET/POST、`render_style` 枚举、`delivery_dir` 无 `..` 路径穿越
- `delivery_dir`：相对 `files/` 下白名单根，或绝对路径但必须在配置允许前缀内（安全）

### 4.3 SimpleReportPlanService（MD 编排）

**输入：** `simpleReportId` + `userPrompt`（可空）  
**输出：** `plan_md` + 新建 `SimpleReportRun`

MD 骨架（固定结构，便于人审与机器校验）：

```markdown
# 简化报告计划：{name}
## 诉求
{userPrompt}

## 1. {block.title}
- 查询目的：{由 prompt_hint + 诉求归纳，可模板化}
- 接口：{METHOD} {url}
- 渲染：{BAR_CHART|TABLE|...}
- 预期数据：labels/values（契约见 §5）

## 2. ...
## 交付
- 目录：{delivery_dir}
```

**MVP 实现策略（降低对大模型依赖）：**

1. **服务端模板生成 MD**（必做）：按 blocks 机械展开，诉求原文写入「诉求」节。  
2. **可选增强**：若后续接入 LLM，仅润色「查询目的」文案，**不得改 url/样式**。  
3. OpenClaw Skill 可先本地改 MD 再 `PUT plan`，但 confirm 前服务端仍以 **配置快照** 为准执行拉数（防篡改接口）。

> 结论：出 Word 的数据路径以 **配置快照** 为准；MD 是给人看的计划与审计材料。

### 4.4 SimpleReportHttpFetcher

| 项 | 约定 |
|----|------|
| 客户端 | `HttpURLConnection` 或 Java 8 可用的轻量封装（与现网一致，不强制引入新 HTTP 栈） |
| 超时 | 连接/读取可配置，默认如 10s / 30s |
| 体积 | 响应体上限（如 2MB），超限失败该块 |
| 协议 | 仅 `http`/`https`；禁止 `file:` 等 |
| 鉴权 | MVP 无 |
| 成功 | HTTP 2xx + JSON |

### 4.5 数据归一化（SimpleReportDataNormalizer）

将接口 JSON → 引擎 `strData` 结构。

**强制入站契约（推荐锁定 Q3）：**

```json
{
  "labels": ["计算机", "机械"],
  "values": [120, 80]
}
```

规则：

- `labels.length == values.length`，且 ≥1
- `values` 为数字或数字字符串
- → `headList` = `[["名称","数值"]]`
- → `tableList` / `chartDataList` 行数据由 labels/values 拉链生成（`chartDataList` 首行表头）

**失败块占位（Q7 默认）：**

- 仍输出一个 TABLE 节点：`tableList=[["错误","拉取失败: {简短原因}"]]`，`strText` 可注明跳过
- Run 级 `warn_count++`；若全部块失败 → Run `FAILED`

### 4.6 SimpleReportAssembleService

对快照 blocks（成功或占位）生成扁平树：

```text
[ TITLE(报告名)
    ├── METRIC(block1)
    ├── METRIC(block2)
    └── ...
]
```

映射：

| `render_style` | `displayType` | `chartStyle` |
|----------------|---------------|--------------|
| `TABLE` | TABLE | null |
| `BAR` | CHART | BAR |
| `PIE` | CHART | PIE |
| `LINE` | CHART | LINE |

`overallSetting`：

- `nMetricsCount` = 实际 METRIC 数
- `dataSetConfigList` = `[]`（静态指标）
- 封面/底图：MVP 可空；若配置有再扩展

### 4.7 SimpleReportRunService（执行与交付）

1. 校验 Run=`PENDING_CONFIRM`  
2. 置 `GENERATING`；分配 `engine_report_id`（§6.4）  
3. 顺序拉数 + 归一化 + 组装  
4. `CreateReportVO`：`reportId=engine_report_id`，`reportName=simpleReport.name`  
5. `SpireReportUtil.createReport(...)`  
6. 成功同步（仿 `ManagedReportGenerateSync`）：  
   - 更新 Run `SUCCESS` + `file_path`  
   - `Files.copy` 到 `delivery_dir / {name}_{runId}_{ts}.docx`（目录不存在则创建；路径已校验）  
   - 记 `delivery_path`  
7. 失败：`FAILED` + `fail_message`；清理 lock（Spire 已有 catch）

### 4.8 Controller API 面（摘要；细则留给 API 设计）

| 方法 | 路径（建议） | 说明 |
|------|----------------|------|
| CRUD | `/simple-report` … | 配置 |
| POST | `/simple-report/{id}/plan` | body: `{ "userPrompt": "..." }` → run + planMd |
| PUT | `/simple-report/runs/{runId}/plan` | 可选：覆盖 planMd 文案（不改执行快照） |
| POST | `/simple-report/runs/{runId}/confirm` | 确认执行 |
| GET | `/simple-report/runs/{runId}` | 状态、planMd、路径 |
| GET | `/simple-report/runs/{runId}/download` | 可选：读 `file_path` 下载 |

响应风格对齐现网：`code` 0 / -1 + `message`。

### 4.9 管理台

- 新 panel：列表、编辑区块表（多行：标题/URL/方法/样式）、交付目录、创建计划、展示 MD、确认生成、看状态  
- 不复用就业学校选择器

### 4.10 OpenClaw Skill（旁路包）

建议目录（实现时再建，本方案只定形态）：

```text
skills/ai-simple-report/SKILL.md
```

Skill 职责：

1. 调用 `GET` 配置 / `POST plan`  
2. 把返回的 `planMd` 展示给用户  
3. **等待用户明确确认**后调 `POST confirm`  
4. 轮询 `GET runs/{id}` 或 Redis 进度说明  
5. **禁止**引导使用 python-docx / Word MCP 作为主路径

## 5. 接口响应契约（锁定建议）

为降低 Agent 与归一化复杂度，MVP **只接受**：

```json
{ "labels": ["string", "..."], "values": [number, "..."] }
```

或兼容包装：

```json
{ "code": 0, "data": { "labels": [], "values": [] } }
```

（`code!=0` 视为该块失败。）

扩展（后期）：`series` 多柱、表格二维 `rows`——不进 MVP。

## 6. 关键设计

### 6.1 reportId / Redis 冲突

`managed_report.id` 与 `simple_report_run.id` 若都当 `reportId` 传入 Spire，会撞 `report:list:%d` / `report:lock:%d`。

**方案（推荐）：** `engine_report_id = SIMPLE_REPORT_ID_BASE + run.id`  
例如 `BASE = 1_000_000_000`，保证与管理台报告 id 段分离；Run 表存 `engine_report_id`。

备选：Redis key 增加前缀（需改 Spire/Constant，侵入较大，不优先）。

### 6.2 路径穿越

- `delivery_dir`、URL host 校验  
- 禁止 `../`；目录必须落在 `application.yml` 配置的 `simple-report.delivery-root` 之下  

### 6.3 SSRF

无鉴权 + 服务端代发 HTTP → **高风险**。MVP 约束：

- 配置项 `simple-report.http.allowed-host-suffixes` 或 IP 白名单  
- 禁止链路本地地址（127.0.0.1、::1、元数据 IP）——**Must**  

### 6.4 密钥

无；邮件后期用环境变量。勿把 Token 写进 Skill 明文（若后期加鉴权，用服务端配置引用）。

## 8. 定时（Quartz）

### 8.1 载体

- 引入 Spring 支持的调度（`spring-boot-starter-quartz` 或 `@Scheduled` + 库表 cron）。  
- 若要「每报告独立 cron」：表字段 `cron_expr` + 动态调度；MVP 可先全局配置 + 报告 `schedule_enabled`。

### 8.2 Job 行为

```text
Quartz 触发 → 对启用定时的 SimpleReport
  → 创建 Run + 模板 MD（诉求=「定时自动」）
  → 策略分支（§8.3）
```

### 8.3 与「必须人工确认」的关系（待产品拍板，方案给默认）

| 策略 | 说明 |
|------|------|
| **A（推荐默认）** | 定时只生成 `PENDING_CONFIRM`，通知人去确认（邮件/列表角标后期） |
| B | 定时直接 `confirm`（跳过确认），与 Q6 对话确认冲突，仅当产品改口 |

PRD Q6 已标「必须人工确认」→ 技术方案默认 **策略 A**。
