# PRD 分析：报告下载

**需求来源：** 一段话 — 报告生成之后支持下载。

**功能短名：** `report_download`  
**状态：** DRAFT — 仅需求  
**关联能力：**  
- 报告管理：`report_manage`（已具备创建 / 生成 / 终态回写 `file_path`）  
- 管理台：`/admin/index.html#report`  
- 既有落盘：生成成功后 `.docx` 写磁盘，路径入 `managed_report.file_path`

**说明：** 本项承接 `report_manage` PRD 中里程碑「体验补齐 · 下载入口」，并**覆盖**原 Q14「仅落盘路径、不提供下载」的默认结论——本期改为**可下载**。

---

## 1. 问题陈述

报告管理已能异步生成 Word，并在成功后把落盘路径写入库表（`file_path`）。管理台列表目前只展示路径文本，**没有可用的下载入口**；运营人员无法在浏览器内直接拿到 `.docx`，只能靠本机路径或运维取文件。

若不做：生成闭环停在「有文件、取不到」，一键生成的业务价值无法落地到交付物。

---

## 2. 目标用户 / JTBD

**Primary：** 管理台配置 / 运营人员（与「报告管理」同一角色）。

**JTBD：**

| 场景 | When | I want | so that |
|------|------|--------|---------|
| 下载成稿 | 报告生成成功后 | 在报告管理里一点即可下载 `.docx` | 立刻交付或本地核对，不必找服务器路径 |
| 失败时不误下 | 生成中或失败时 | 看不到可用下载 / 点了有明确提示 | 不会下到空文件或旧稿而不自知 |

**Not for（本期默认）：** 学生端 / 对外门户自助下载；邮件推送附件；批量打包 ZIP；短链分享给校外。

---

## 3. Evidence

Assumption — 来自产品口述「生成之后支持下载」及现状观察：列表已展示 `filePath` 字符串但无下载动作；`report_manage` 原 Q14 仅落盘、未做入口。尚无外部工单统计。建议上线后用「成功态报告的下载次数 / 成功后 24h 内是否发生下载」验证。

---

## 4. 成功标准

| 指标 | 目标 | 如何衡量 |
|------|------|----------|
| 成功可下 | Must | `generateStatus=成功` 且存在有效文件时，管理台可触发下载并得到 `.docx` |
| 文件正确 | Must | 下载内容与本次成功生成的报告一致（非空、可打开） |
| 非成功不可下 | Must | 草稿 / 生成中 / 失败（且无历史成功文件时）不可下载或明确拒绝 |
| 失败可读 | Must | 文件缺失、权限、状态不符时有可读错误提示 |
| 体验可达 | Should | 列表或详情上有明显「下载」入口，无需复制服务器路径 |

---

## 5. 方案概述（What）

```text
报告下载
├── 前置：报告实例生成成功，系统已持有落盘 .docx（file_path）
├── 入口：管理台「报告管理」成功态提供下载
└── 行为：按报告实例取文件 → 浏览器下载（建议带可读文件名）
```

**边界：**  
- 本功能负责「生成成功后的取件 / 下载体验」；  
- **不重做** Word 生成与指标聚合；  
- **不新建**独立「下载中心」产品（后置，见 Won't）。

---

## 6. 功能需求

### 6.1 下载能力（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| D-1 | 生成**成功**后，可对报告实例发起下载 | Must | 成功判定与 `generateStatus` / 有有效文件对齐 |
| D-2 | 下载产物为该实例最近一次成功生成的 `.docx` | Must | 与库内记录的成功文件一致 |
| D-3 | 下载时使用可读文件名（建议含报告名称） | Should | 非法文件名字符需清理；见 Q3 |
| D-4 | 管理台提供「下载」操作入口 | Must | 列表和/或详情；仅成功（或见 Q2）时可用 |
| D-5 | 文件不存在或路径失效时失败可读 | Must | 不返回空流；提示重新生成或联系运维 |
| D-6 | 未授权访问防护 | Must | 至少：仅能按报告实例 ID 取本系统已登记文件；禁止任意路径穿越（见 Q5） |

### 6.2 与生成状态的关系（Must）

| ID | 需求 | 优先级 | 备注 |
|----|------|--------|------|
| S-1 | 草稿、生成中：不提供下载或下载被拒绝 | Must | |
| S-2 | 失败：默认不可下载 | Must | 若仍保留「上一次成功文件」是否允许下载见 **Q2** |
| S-3 | 再次生成成功后，下载指向**新**成功文件 | Must | 与现有「成功覆盖 `file_path`」一致 |
| S-4 | 改学校/模版后未重生成：下载仍可能对应旧稿 | Should | UI 提示「文件对应上次成功生成」——与 `report_manage` 风险一致 |

### 6.3 明确不做（Won't / 后置）

| 项 | 原因 |
|----|------|
| 学生 / 校外门户下载 | 非管理台角色 |
| 邮件 / IM 推送附件 | 另立通知能力 |
| 批量多报告 ZIP | 可后置 |
| 在线预览 Word | 成本高，本期只要下载 |
| 永久外链 / 无鉴权 CDN | 安全与运维成本 |
| 历史版本多文件存档 | 当前模型仅保留最近成功路径 |

---

## 7. Hypothesis

We believe **「生成成功后在管理台一键下载 `.docx`」** will **打通从生成到交付** for **管理台运营人员**.  
We'll know we're right when **成功态报告可在管理台直接下载到可打开的 Word，且无需登录服务器取路径**.

| # | 里程碑 | 用户可见结果 | 状态 |
|---|--------|--------------|------|

## 9. 待确认

- [管理台 + HTTP 下载接口] **Q1** 下载形态：仅管理台按钮调后端流式下载，还是同时给外部系统稳定下载 API？  
- [仅成功态可下；失败保留旧文件时可下旧稿] **Q2** `generateStatus=失败` 但库中仍有上次成功 `file_path` 时：是否仍允许下载旧文件？  
- [报告名 + `.docx`] **Q3** 下载文件名规则：固定 `报告名.docx` / 加时间戳 / 加学校名？  
- [同步流式下载] **Q4** 大文件：同步流式即可，还是需要异步打包/短时签名 URL？（MVP 建议同步流式）  
- [禁止按裸路径下载] **Q5** 是否禁止客户端直接使用 `filePath` 字符串访问磁盘；一律经报告 ID 由服务端解析？  
- [本期不做登录] **Q6** 管理台当前无登录时，下载是否与现有 admin 同级「内网可信」？是否需后续加鉴权？  
- [软删后不可下] **Q7** 报告软删后：下载接口是否一律 404/拒绝？（建议：拒绝）  
- [不计入新表] **Q8** 是否需要下载审计日志（谁、何时、哪份报告）？本期默认不做。

（括号内为建议默认，审查时可改；未拍板前实现侧按建议或标 TBD。）

## 10. 风险

| 风险 | 可能性 | 影响 | 缓解 |
|------|--------|------|------|
| `file_path` 与磁盘不一致（被删、迁移） | 中 | 下载失败 | D-5 可读错误；引导重新生成 |
| 改配后下到旧稿 | 中 | 业务误用旧数据 | S-4 提示；或强制改配后禁用下载直至重生成（见 Q2 延伸） |
| 无鉴权环境下被扫 ID 拖库 | 中（视部署） | 报告泄露 | Q6；内网部署说明；后续鉴权 |
| 路径穿越 / 任意文件读 | 低（若实现不当则高） | 严重安全事故 | Q5：只按报告 ID + 库内路径校验 |
| 文件名中文 / 特殊字符 | 低 | 浏览器乱码 | Q3 清理与 Content-Disposition 约定 |

## 11. 与现有能力的关系

| 已有 | 本 PRD 增量 |
|------|-------------|
| 生成成功写 `file_path` | 提供基于报告实例的下载能力 |
| 列表展示 `filePath` 文本 | 增加「下载」操作，弱化「抄路径」 |
| Redis 进度 / 锁 | **不替代**下载；下载只看终态与文件 |
| 模版图片 `/files/report-template/**` | 报告 docx 下载宜独立约定，勿与模版静态资源混用任意路径 |
